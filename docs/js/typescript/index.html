타입 호환성

Section titled “타입 호환성”

TypeScript에서는 타입 이름이 달라도, 필요한 속성과 메서드를 가지고 있으면 같은 타입처럼 사용할 수 있다.

위의 그림은 typescript 호환성에 대한 toss 예시이다.

만약 버거의 타입이 Food와 정확하게 일치한다면 실제 정상처럼 동작할 수 있는 코드가 된다.
그렇다면 타입 시스템은 개발자의 의도에 맞게 유연하게 대응 하는것이 좋을수도 있다. 이게 바로 TypeScript의 타입 호환성이다.

Naver의 설명

Section titled “Naver의 설명”

Typescript의 타입 검사란 어떤 심벌에 대한 각종 대입/참조/연산이 가능한지 확인하는 과정이라고 한다.
구조적 맥락 아래에서 가능한 타입을 추론하는데 그 과정에서 너무 많은 조합이 생길 수 있어서 그리디한 알고리즘으로 구현해서 타입을 추론하는 것 같다고 이야기한다.
여기서 제약 충족 문제(Constraint Satisfaction Problem)라는 개념도 나오고 NP-완전 문제도 나온다.

자세히 설명하자면,
제약 충족 문제는

이렇게 여러 조건을 동시에 만족하는 답을 찾는 문제를 제약 충족 문제라고 한다.
하지만 여러가지 조건을 동시에 찾는 NP-완전 을 해결하는 것이 불가능하기 때문에 그리디한 알고리즘으로 구현해서 타입을 추론하는 것 같다고 이야기 하는 것 같다.

Typescript에서는 유연성만을 생각 할 순 없다.
따라서 어떤 경우에 호환이 되고 안되는지를 규칙이 필요하다.
그 규칙을 만드는 방법은 명목적 서브타이핑(Nominal Subtyping)과 구조적 타이핑(Structural Typing)이 존재한다.
