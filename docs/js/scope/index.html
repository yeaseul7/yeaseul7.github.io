scope 란?

Section titled “scope 란?”

값(변수)과 “표현식”이 현재 실행되는 컨텍스트를 의미한다.
만약 변수나 표현식이 해당 스코프 안에 있지 않다면 사용할 수 없다.
scope는 포함 구조를 가진다.
즉 상위 크로프가 하위 스코프를 포함함에 따라서 하위 -> 상위로는 접근이 가능하지만 상위 -> 하위로는 접근이 불가능하다.

scope의 종류

Section titled “scope의 종류”
  • 전역 스코프 : 어디서든 접근 가능한 스코프
  • 함수 스코프 : 함수 내부에서만 접근 가능한 스코프 -> function으로 생성된 범위
  • 블록 스코프 : 블록 내부에서만 접근 가능한 스코프
  • 모듈 스코프 : 모듈 내부에서만 접근 가능한 스코프

Section titled “함수 스코프 var과 블록스코프가 비동기 함수를 만났을때 일어나는 일”

위의 결과는 5 5 5 5 5이다.

var는 블록 스코프가 아니고 함수 스코프다. 이 의미는 곧

var는 for안에 안묶인다. 함수 스코프 이기 때문에
setTimout이 1초 뒤에 실행되는 시점에는 이미 i가 5가 되어버린 상태이다. 따라서 5가 5번 찍히는 것이다.

반면에 let은 어떨까? let은 블록 스코프이다. 즉, for문 안에 묶인다.

따라서 for안에서 반복될때마다 새로운 렉시컬 환경을 만든다.
1번째 반복의 i = 0 └── 첫 번째 콜백이 이 i를 기억

2번째 반복의 i = 1 └── 두 번째 콜백이 이 i를 기억

3번째 반복의 i = 2 └── 세 번째 콜백이 이 i를 기억

내 답변
let도 호이스팅 과정을 거칩니다. let은 호이스팅으로 인해 먼저 메모리에 등록은 되지만 선언문까지 가는 동안에 공백이 발생합니다. 이걸 TDZ라고 하는데 let은 블록 스코프의 특징을 가짐으로 해당 스코프 안에서만 유효하다는 장점이 존재합니다. 이 TDZ가 존재하지 않는다면 이 특색이 무색해질 것입니다.

내 답변

먼저 코드를 토큰화하면 함수 표현식은 빈 변수를 먼저 선언하고 후에 함수를 집어 넣는 방식입니다. var를 사용하더라도 함수 자체가 호이스팅 되는것이 아니고 함수를 넣을 빈 변수만 호이스팅됩니다. 따라서 표현식의 렉시컬 환경은 빈 변수만 생성하고 초기화 이후에 함수를 할당하기 때문에 바로 호출이 불가능합니다

하지만 함수 선언문은 함수 자체를 먼저 생성하고 초기화 하기 때문에 코드 상단에서 바로 호출 가능합니다.

피드백: 아주 정확해! 선언문은 평가 단계에서 환경 레코드에 함수 객체까지 생성해서 등록해버리기 때문에 즉시 호출 가능한 거야. 반면 표현식은 변수 식별자만 등록되고, 실제 함수가 할당되는 것은 런타임(코드 실행 단계)에 이루어지므로, 그전까지는 변수 이름만 알고 있을 뿐 함수라는 사실을 모르는 상태라는 점을 강조하면 최고야.

3. “작성하신 inner() 예제에서 inner 함수가 outer의 x를 참조하여 값을 출력하는 것은 클로저 현상입니다. 여기서 의문입니다. outer 함수가 호출을 마치고 실행 컨텍스트가 종료되면, outer의 렉시컬 환경도 소멸해야 하는 것이 원칙 아닌가요? 자바스크립트 엔진은 inner 함수가 존재하는 한 outer의 렉시컬 환경을 어떻게 보호(유지)하며, 이는 메모리 관리 측면에서 어떤 주의점을 시사하나요?”

내 답변
클로저는 x의 값을 복사하는것이 아닌 대상을 가지고 있습니다. 즉, 계속해서 참조하고 있다는 의미입니다. inner함수가 outer의 x를 계속 참조하고 있다면 실행컨텍스트가 종료되어도 GC가 수집하지 않기 때문에 메모리 관리 측면에서 참조를 끊어 줄 필요가 있습니다.

3번 질문: 클로저와 메모리 관리
평가: 90점

피드백: 핵심인 ‘참조 유지’를 잘 설명했어. 엔진이 outer의 렉시컬 환경을 유지하는 이유를 더 기술적으로 풀자면, inner 함수의 [[Environment]] 내부 슬롯이 outer의 렉시컬 환경을 참조하고 있기 때문이야. GC(가비지 컬렉터)는 “어떤 스코프에서 이 변수를 참조 중인가?”를 보는데, inner가 살아있으니 outer의 환경도 계속 참조되는(Reachable) 상태로 남는 거지.

주의점: 무조건 참조를 끊는 게 답은 아니야. 오히려 클로저를 활용해 ‘데이터를 은닉’하고 ‘상태를 보존’하는 것은 아주 의도된 프로그래밍 패턴이니까, “필요 없는 경우에만 클로저를 의도치 않게 남발하지 않도록 주의한다”라고 덧붙이면 훨씬 성숙한 답변이 돼.
