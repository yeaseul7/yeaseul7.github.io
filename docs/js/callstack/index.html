callstack

callstack

콜 스택은 JavaScript 엔진이 현재 어떤 함수가 실행 중인지 함수 실행이 끝난 뒤 어디로 돌아가야 하는지를 관리하기 위해 사용하는 스택 구조이다.

JavaScript에서 함수가 호출되면 해당 함수의 실행 컨텍스트가 콜 스택에 쌓인다.
함수 실행이 끝나면 해당 실행 컨텍스트는 콜 스택에서 제거되고 이전 실행 컨텍스트로 돌아간다.

콜 스택은 LIFO, 즉 Last In First Out 구조를 가진다.
가장 마지막에 호출된 함수가 가장 먼저 실행을 끝내고 자신을 호출했던 위치로 돌아가야 하기 때문이다.

그 이유를 설명해보면

위 코드에서 drawSquare 함수가 실행되는 도중 drawLine 함수가 호출된다.
drawLine 함수가 실행되는 동안에는 drawSquare의 실행이 잠시 멈춘다.
drawLine 함수가 끝나면 다시 drawSquare 함수의 다음 줄로 돌아와야 한다.

함수 호출은 가장 안쪽에서 호출된 함수부터 먼저 끝나야 한다.
이 흐름이 스택의 LIFO 구조와 잘 맞기 때문에 JavaScript 엔진은 콜 스택을 사용한다.

콜 스택과 실행 컨텍스트

Section titled “콜 스택과 실행 컨텍스트”

JavaScript에서 함수가 호출되면 실행 컨텍스트가 생성된다.
실행 컨텍스트는 현재 실행 중인 코드가 어떤 환경에서 실행되는지를 나타내는 정보이다.

실행 컨텍스트에는 대략 다음과 같은 정보가 포함된다.

  • 현재 실행 중인 코드의 위치
  • 변수와 함수 선언에 대한 정보
  • this 바인딩
  • 외부 렉시컬 환경에 대한 참조

함수가 호출되면 실행 컨텍스트가 콜 스택에 쌓이고, 함수 실행이 끝나면 콜 스택에서 제거된다.

콜 스택과 메모리 관리

Section titled “콜 스택과 메모리 관리”

콜 스택은 함수 호출과 종료 순서가 명확하기 때문에 메모리를 빠르게 관리할 수 있다. 함수가 호출되면 필요한 정보가 스택에 쌓이고 함수가 끝나면 위에서부터 제거된다.

이 과정은 push와 pop 방식으로 이루어지기 때문에 비교적 단순하고 빠르다.

하지만 힙은 객체나 배열처럼 수명이 일정하지 않은 데이터를 저장하는 공간이다. 어떤 객체는 함수가 끝난 뒤에도 다른 곳에서 참조될 수 있기 때문에, 스택처럼 단순히 위에서부터 제거할 수 없다.

위 코드에서 createUser 함수의 실행 컨텍스트는 함수가 끝나면 콜 스택에서 제거된다.
하지만 user 객체는 result가 참조하고 있기 때문에 바로 사라지지 않는다.

함수 실행 자체는 콜 스택에서 관리되지만 객체처럼 함수 밖에서도 사용될 수 있는 데이터는 힙에 남을 수 있다.

재귀와 스택 오버플로

Section titled “재귀와 스택 오버플로”

재귀 함수는 함수 내부에서 자기 자신을 다시 호출하는 함수이다.

재귀 함수가 호출될 때마다 새로운 실행 컨텍스트가 콜 스택에 쌓인다. 재귀의 깊이가 너무 깊어지면 콜 스택의 한계를 초과할 수 있다.

이때 발생하는 오류가 스택 오버플로이다.

위 코드는 종료 조건 없이 자기 자신을 계속 호출한다. 그 결과 콜 스택에 실행 컨텍스트가 계속 쌓이고, 결국 콜 스택의 한계를 넘게 된다.

그래서 깊이가 예측되지 않는 재귀는 반복문이나 명시적인 자료구조로 바꾸는 것이 더 안전할 수 있다.

함수 안에 함수가 있을 때 변수는 어떻게 찾을까?

Section titled “함수 안에 함수가 있을 때 변수는 어떻게 찾을까?”

JavaScript는 변수를 찾을 때 현재 스코프에서 시작해 바깥 스코프로 이동하며 변수를 탐색한다.
이 구조를 스코프 체인이라고 한다.

중첩 함수의 스코프 탐색 (스코프 체인)

Section titled “중첩 함수의 스코프 탐색 (스코프 체인)”
  1. inner 스코프에서 name 찾기
  2. 없으면 outer 스코프에서 name 찾기
  3. 없으면 전역 스코프에서 name 찾기

함수 안에 함수가 선언되면 안쪽 함수는 자신이 선언된 위치의 외부 렉시컬 환경을 참조할 수 있다.
JavaScript는 변수를 찾을 때 현재 스코프에서 바깥 스코프로 스코프 체인을 따라 탐색한다.
이 구조 덕분에 내부 함수는 외부 함수가 종료된 뒤에도 필요한 변수를 기억할 수 있다. -> Closure

스택 프레임이 존재하는 이유

Section titled “스택 프레임이 존재하는 이유”

함수가 동작할때마다 별도의 스택 프레임이 없다면, 변수가 뒤섞여서 js에서 스코프 체인이 작동할 수 없다.

최근 JavaScript 엔진은 비동기 흐름을 더 잘 추적하기 위해 async stack trace를 제공하기도 한다.

Ai 질문

Section titled “Ai 질문”

1. “콜 스택이 꽉 차면(스택 오버플로) 프로그램은 왜 멈추나요?”

Section titled “1. “콜 스택이 꽉 차면(스택 오버플로) 프로그램은 왜 멈추나요?””
  • 콜스택은 한정된 메모리이기 때문에 한계가 있습니다. 빠르게 넣고 뺄 수 있는 만큼 크기가 작습니다.

AI 피드백
아주 정확해! 여기에 ‘에러 발생 시 엔진의 동작’을 한 문장 추가하면 완벽해.
“메모리가 넘치면 자바스크립트 엔진은 프로그램이 더 이상 안정적으로 실행될 수 없다고 판단하여, ‘Stack Overflow’라는 에러를 던지고 실행을 강제로 중단(Panic)시킵니다.”라고 덧붙여봐.

점수: 90/100

2. “만약 함수가 실행되는 도중 에러가 나면, 콜 스택에 쌓여 있던 정보들은 어떻게 되나요?”

Section titled “2. “만약 함수가 실행되는 도중 에러가 나면, 콜 스택에 쌓여 있던 정보들은 어떻게 되나요?””
  • 콜스택에 쌓여있던 정보들을 추적하여 stack을 보여줍니다. (error stack trace)

AI 피드백
이 과정을 전문 용어로 ‘스택 언와인딩(Stack Unwinding)‘이라고 불러. “에러가 발생하면 콜 스택은 에러가 난 지점부터 시작해서, 호출했던 상위 함수들을 하나씩 제거(Pop)하며 에러를 전파합니다. 이때, 개발자에게 에러가 발생한 경로를 알려주기 위해 현재 스택 상태를 기록하는데, 이게 바로 ‘Stack Trace’입니다.”라고 답하면 정말 시니어 같을 거야.

점수: 95/100

3. “함수 안에 다른 함수가 없고, 그냥 코드 한 줄씩 차례대로 실행된다면 콜 스택이 꼭 필요할까요?”

Section titled “3. “함수 안에 다른 함수가 없고, 그냥 코드 한 줄씩 차례대로 실행된다면 콜 스택이 꼭 필요할까요?””
  • 아니요 한줄씩 실행된다면 그 다음 차례를 예측할 수 있기때문에 필요 없습니다. 콜스택은 중첩된 함수에서 실행이 끝나면 자신이 돌아갈 곳을 찾는 편하고 빠른 메커니즘이기때문에 유리한것 입니다.

AI 피드백
완벽해! “복귀 주소(Return Address)“라는 개념을 짚어낸 게 정말 좋아. 함수 호출이 중첩되지 않는다면 단순히 프로그램 카운터(PC)만 다음 줄로 옮기면 되지만, 함수 호출은 ‘나갔다가 다시 돌아와야 하는 문맥(Context)‘이 생기기 때문에 그 문맥을 저장할 ‘가방’이 필요한 거거든. 그 가방이 바로 스택이라는 걸 아주 잘 이해했어.

점수: 100/100
