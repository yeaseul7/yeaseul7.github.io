🫨 fetch API란?

Section titled “🫨 fetch API란?”

fetch의 사전적 의미 : (무엇을) 가져오다, (무엇을) 얻다
브라우저에 기본으로 내장된 자바스크립트 전역 메서드(함수)

과거에는 XMLHttpRequest를 사용하거나 AJAX나 axios같은 무거운 라이브러리를 사용해야했다.
하지만 최근에는 웹브라우저에 표준으로 fetch가 내장되어 네트워크 통신이 간편해졌다.
(2017~2018년부터 대부분의 브라우저)

😎 fetch의 특징

Section titled “😎 fetch의 특징”
  • Promise 기반이다.
  • 404나 500 같은 에러가 발생해도 reponse를 뱉었다고 간주하기 때문에 catch로 잡히지 않는다. - Resolve로 처리한다.
    따라서 response.ok 나 response.status로 에러를 체크해야한다.

😯 여기서 Worker란?

Section titled “😯 여기서 Worker란?”

Window 컨텍스트: DOM을 조작하는 메인 화면 환경
Worker 컨텍스트: 무거운 연산이나 데이터 처리를 전담하는 백그라운드 환경

브라우저는 기본적으로 싱글스레드이다.
DOM을 조작하는데에 바쁜 Window 컨텍스트에서 무거운 연산이 발생하면 UI가 멈추는 현상이 발생할 수 있다.

fetch()는 Window와 Worker 컨텍스트 양쪽에서 모두 사용할 수 있는 전역 메서드라는 점을 고려한다면,
Worker 컨텍스트를 사용해서 백그라운드에서 네트워크 요청을 보낼 수 있다.

쓰레드를 여러개 사용하는 것은 비용 측면에서 반드시 옳다고 할 수 없습니다. 무거운 연산이 포함된 fetch 요청이 아니라면 즉시 main thread에서 DOM에 빠르게 반영하는 것이 더 효율적일 수 있기 때문입니다. 만약 간단한 연산에 의해 즉각 DOM의 변화가 필요한 상황이라면 window 객체에서 직접 fetch 요청을 실행 할수도 있다고 생각합니다.

ai 피드백

Section titled “ai 피드백”
  • postMessage 통신 오버헤드(비용)와 메모리 격리 구조에 대한 언급
  • Window와 Worker는 메모리 공간이 완전히 격리되어 있어 데이터를 주고받을 때 postMessage API를 통한 ‘구조화된 복제(Structured Clone)’ 프로세스가 일어납니다. 이 과정에서 데이터의 직렬화/역직렬화 및 메모리 복사 비용이 발생하므로, 무거운 연산이 없는 가벼운 fetch 요청까지 모두 Worker로 넘기면 오히려 이 통신 비용 때문에 전체적인 레이턴시가 늘어날 수 있습니다.

🫨 웹(Web)을 지탱하는 4대 발명품

Section titled “🫨 웹(Web)을 지탱하는 4대 발명품”
  • HTML (그릇): 정보를 하이퍼텍스트(링크) 형태로 담아두는 문서 형식.
  • HTTP (규격): HTML을 컴퓨터끼리 오차 없이 주고받기 위해 만든 통신 규약.
  • WorldWideWeb (클라이언트): 문서를 화면에 그리고 사용자에게 보여주는 최초의 브라우저.
  • httpd (서버): 누군가 요청하면 하이퍼텍스트 문서를 꺼내서 던져주는 최초의 웹 서버 프로그램.

HTTP/0.9 - 원라인 프로토콜

Section titled “HTTP/0.9 - 원라인 프로토콜”

한줄만 주고 받아서 원라인 프로토콜

특징

Section titled “특징”
  • header가 없으니까 content-type, content-length 같은 정보가 없다.
  • states code도 없다. (200 OK, 404 Not Found 같은 코드가 없다.)
  • get 요청만 지원한다. (post, put, delete 같은 다른 메서드가 없다.)

HTTP/1.0

Section titled “HTTP/1.0”
  • 버전 정보를 포함 (HTTP/1.0)
  • HTTP 헤더 도입 (Content-Type, Content-Length, User-Agent 등)
  • 상태 코드 도입 (200 OK, 404 Not Found 등)

⚠️ 한계

  • 연결마다 TCP 3-way Handshake를 해야해서 연결 설정이 느리다. (Connection Overhead)

HTTP/1.1

Section titled “HTTP/1.1”
  • keep-alive 연결 도입 (연결 재사용)
  • 파이프라이닝 지원 (여러 요청을 한 번에 보내고 응답을 기다리지 않고 처리)
  • Rest API의 등장 (HTTP 메서드: GET, POST, PUT, DELETE 등)
  • CORS (Cross-Origin Resource Sharing) 도입

⚠️ 한계 Application HOLB

  • 요청은 연속적으로 보낼 수 있지만 응답은 순차적으로 돌아온다. (Head of Line Blocking)

HTTP/2

Section titled “HTTP/2”
  • 구글의 SPDY 프로토콜을 기반
  • 텍스트 형식의 HTTP/1.x에서 바이너리 형식으로 변경 (Binary Protocol)
  • 멀티플렉싱 지원 (한 연결에서 여러 요청과 응답을 동시에 처리)
  • 헤더 압축 (Header Compression) -> 중복 헤더 압축

⚠️ 한계 Transport(TCP) HOLB TCP는 데이터 유실이 나면 무조건 멈춰서 재전송을 기다림
즉 패킷 하나가 유실되면 유실되지 않은 다른 스트림까지 전송 계층에서 멈춰버리는 현상

HTTP/3

Section titled “HTTP/3”
  • 멀티플렉싱을 지원하되, 각 요청을 독립된 스트림으로 처리하는 QUIC 프로토콜 기반
  • TCP처럼 연결할 때 여러 번 Handshake하지 않아서 연결 설정 속도(0-RTT)가 극도로 빠르다.

단하나의 커넥션이 위험 한 것은 터널을 예시로 생각하면 이유가 명확합니다. 아무리 많은 짐을 효율적으로 담은 트럭들을 보낸다고 한들 하나의 터널에서 앞에 길을 막아버리면 뒤의 방대한 짐을 가지고 있는 트럭이 움직이지 못하고 도착지로 도달하지 못하기 때문입니다. HTTP/1.1은 요청마다 커넥션을 만들기 때문에 연결의 끊김을 판별하고 처리하기에 HTTP/2보다 유연한 구조라고 볼 수 있습니다. HTTP/3는 이 문제를 해결하기 위해 아예 터널을 여러개 뚫어버린 상황입니다. 따라서 다른 통신들에 영향을 받지 않기 때문에 이를 완벽하게 극복했다고 볼 수 있습니다.
