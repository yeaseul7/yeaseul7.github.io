프로젝트를 Vercel로 배포해 오다가, 구조가 잡힐수록 **요금 정책**이 눈에 들어왔다. 같은 고민을 하는 사람이 있을 것 같아, 왜 그렇게 느꼈는지와 Cloudflare로 넘어가면서 맞닥뜨린 제약, 그리고 실제로 어떻게 풀었는지를 짧게 정리했다.

### Vercel이 부담스럽게 느껴지는 이유

Vercel의 기본 축은 **Serverless Functions**에 가깝고, Node.js 친화적인 실행 환경을 전제로 한다. Buffer나 Node 전용 SDK처럼 **호환성과 개발 편의**를 넓히려면 그만큼 무거운 런타임을 끌고 가야 하고, 그 비용이 요금에 반영된다고 이해했다. “비싸다”는 말은 단순히 단가만의 문제가 아니라, **어떤 추상화를 사느냐**에 가깝다.

### Cloudflare는 왜 다른가

Cloudflare Workers 쪽은 **Edge Runtime(V8 Isolates)** 를 택했다. 무거운 Node.js 전체를 들고 가기보다 **필요한 V8 격리 환경**만 돌리기 때문에, 같은 트래픽이라도 서버 쪽 리소스 효율이 좋다는 인상을 받았다. 다만 이건 “무조건 싸다”가 아니라 **제약과 트레이드오프가 다르다**는 쪽에 가깝다.

### 3 MiB 번들 제한과 AI

무료(또는 해당 플랜) 기준으로 **Worker 번들 크기에 상한(대략 3 MiB)** 이 있다. AI 기능이 들어가면 의존성만으로도 이 한도를 넘기기 쉽다. Edge에 **무거운 AI 스택을 그대로 얹는 방향**은 현실적으로 막히는 경우가 많다고 보면 된다.

### “그럼 백엔드로 빼면 Firestore는?”

당연히 다음 질문은 이거였다. **무거운 모듈을 백엔드 호출로 빼면**, 데이터는 여전히 Firestore 같은 managed DB에 두게 되고, **읽기·쓰기 횟수에 따라 과금**이 불어나지 않을까.

답은 “설계로 줄인다”에 가깝다. 예를 들어 **Cloud Storage에 메타데이터를 두고 캐싱하는 방식**으로 Firestore 읽기를 줄이면, 비용과 동기화 안정성을 동시에 노릴 수 있다. 숫자로 말하면, 우리 케이스에서는 Firestore 읽기 비용을 **기존 대비 약 95%까지** 줄이는 쪽으로 맞췄다.

### 실제 아키텍처에서 한 선택

**Firebase Functions** 같은 서버리스에는 **메모리·타임아웃** 같은 물리적 한계가 있고, Edge에는 **번들 크기** 한계가 있다. 둘 사이에서 **중량 AI 로직은 Google Cloud Run(컨테이너)** 으로 빼서 마이크로서비스처럼 두었다. Node에 강하게 묶인 AI SDK를 프론트·Edge 쪽에서 걷어내고, **Cloud Run의 REST API**만 호출하게 바꾸니 Cloudflare Edge 배포가 가능해졌고, 번들 크기는 **대략 90% 수준**까지 줄었다.

데이터 쪽은 **Data API**로 생산성을 올리되, **RLS(Row Level Security)** 를 강제해 접근을 테이블 단위로 통제했다. 인프라 설정에서 **모든 테이블을 자동으로 노출하지 않도록** 막아 **최소 권한 원칙**을 지키고, **Cloud Run과 프론트엔드가 쓰는 키를 분리**해 백엔드와 클라이언트의 권한 경계를 명확히 했다.

### 마무리

Vercel에서 Cloudflare로 “옮긴다”는 건 호스트만 바꾸는 일이 아니라, **어디까지를 Edge에 둘지·어디부터를 컨테이너·DB에 둘지**를 다시 그리는 일에 가깝다. 제한이 보이면 그때가 설계를 고치기 좋은 타이밍인 것 같다.
