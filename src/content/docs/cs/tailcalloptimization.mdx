재귀 함수가 스택 메모리를 낭비하지 않도록 컴파일러가 마지막 호출을 반복문 형태로 변환해서 스택 프레임을 재사용하게 만드는 것

#### 재귀함수의 장점
- 코드의 가독성과 간결함
- Tree나 Graph 같은 자료구조는 재귀적인 구조를 가지고 있다. 그래서 이때 사용하면 좋다.
- 재귀는 함수 호출 시마다 stack Frame에 독립적인 지역 변수를 가져서 안전하게 상태를 분리하기 좋다.

#### 재귀 함수의 단점
stack Frame에 의한 stack Overflow
함수 호출과 복귀의 비용이 크다. (ex. context switching, frame 할당..)

### 재귀 함수
```javascript
function sum(n) {
    let s = 0;
    for (let i = 1; i <= n; i++) {
        s += i;
    }
    return s;
}
```
위의 경우는 반복문으로 함수 호출은 처음에 sum을 호출할때만 일어난다. 

```javascript
function sum(n) {
    if (n <= 0) {
        return 0;
    } else {
        return sum(n - 1) + n; // 호출 후 덧셈 연산 대기
    }
}
```

sum(5)를 호출한다고 생각하면
```
sum(5)
  sum(4)
    sum(3)
      sum(2)
        sum(1)
          sum(0)
```
이렇게 `stack Frame`이 생긴다.

위의 일반 재귀함수는 함수 호출 후 남은 연산을 수행해야 하므로 돌아갈 위치를 스택 프레임에 저장해두어야 한다.

#### Tail Call Optimization(Tail Call Optimization, TCO)

함수의 마지막 작업이 다른 함수를 호출하는 것이라면 현재 함수의 스택 프레임을 유지하지 않고 재사용하거나 제거한다.

위의 sum함수를 생각해보면
<pre>
sum(5) frame
sum(4) frame
sum(3) frame
sum(2) frame
sum(1) frame
sum(0) frame
</pre>
이렇게 된다. 

꼬리 최적화하면 <br/>
sum frame 하나만 재사용<br/>
n과 s 값만 계속 갱신

예를 하나만 더 들어보자
```javascript
function sumTail(n, s = 0) {
    if (n <= 0) {
        return s;
    } else {
        return sumTail(n - 1, s + n); // 호출이 마지막 작업!
    }
}
```
위의 함수를
```javascript
function sumLoop(n, s = 0) {
    while (n > 0) {
        s = s + n;
        n = n - 1;
    }
    return s;
}
```
이렇게 바꾸는것과 똑같다.

sum함수를 한번만 호출하고 내부적으로 변수만 업데이트한다. 

### TCO는 항상 일어나나?

아니다. <br/>
언어와 컴파일러, 실행 환경에 따라 다르다.

javascript는 ES6 Strict Mode에서 표준화되었지만 실제로는 제한적으로 적용된다고 한다. 
Chrome, Node.js (V8) 엔진은 한때 지원했으나 Stack Trace를 어렵게 만든다는 이유로 현재는 지원하지 않는다고 한다.

따라서 JavaScript에서 깊은 재귀를 사용할 때는 반드시 스택 오버플로우를 고려해서 반복문으로 리팩토링하는 습관이 중요하다.

### 💡 TCO와 재귀에 관한 심화 질문 3가지
#### 1. "스택 트레이스(Stack Trace)의 디버깅 편의성"과 "TCO"는 왜 양립하기 어려운가요?
stack trace는 함수 호출이 일어난 진입접을 기억하면서 되돌아가며 디버깅을 해주는것인데 TCO는 해당 stack Frame 생성을 생략해서 메모리 효율성을 높이기 때문에 양립하기 어렵다.

*평가: 95점*

피드백: 아주 정확해! "스택 프레임 생략 = 메모리 효율 증가 = 호출 이력 정보(Stack Trace) 소실"이라는 논리가 완벽해. 에러가 났을 때 어떤 함수를 거쳐서 왔는지 보여주는 그 정보가, 프레임이 재사용되면서 사라지니까 디버깅이 어려워진다는 점을 잘 짚었어.

#### 2. TCO가 지원되는 환경이라 할지라도, 재귀 함수는 결국 '함수 호출'을 동반하는데, 반복문에 비해 오버헤드가 완전히 0이 될 수 있을까요?
오버헤드가 0이 될 수 있냐는 말 자체가 이해가 안되긴 하지만 맥락상 이해하자면 0이 될수는 없다 줄어드는 것일 뿐 왜냐면 최초에 한번은 무조건 호출되어야 하기 때문에 그리고 재귀함수가 어떻게 생겼냐에 따라 또 달라질수도 있다.

*평가: 85점*

피드백: "0이 될 수는 없다"는 대답은 기술적으로 정답이야.

보완 포인트: 0이 되지 않는 이유를 '함수 호출 규약(Calling Convention)' 관점에서 설명하면 더 좋아. TCO는 스택 프레임을 '재사용'할 뿐이지, '인자를 전달하고 제어 흐름을 넘기는 과정' 자체는 함수 호출의 비용을 동반하거든. 즉, "프레임 할당은 줄일 수 있지만, 점프(Jump) 명령과 레지스터 설정 비용은 피할 수 없다"고 답변하면 만점이야!


#### 3. 만약 TCO가 지원되지 않는 환경(Node.js 등)에서 재귀 알고리즘을 사용해야만 한다면, 스택 오버플로우를 피할 수 있는 다른 설계 패턴은 무엇이 있을까요?
시간복잡도를 고려해서 설계하자면 일단 stack trace가 필요없는 부분은 최대한 반복문을 사용하고 stack trace가 필요하거나 재귀 알고리즘으로 구현했던 근거가 되는 부분에 사용해서 최대한 방지할 것 같다. 
전문 용어는 모르겠다.

평가: 60점

피드백: "필요 없는 부분은 반복문을 사용하고 재귀가 필요할 때만 사용한다"는 답변은 현실적인 실무 지침이지만, 면접관이 듣고 싶어 하는 '설계 패턴'에 대한 답변으로는 조금 부족해.  

보완 포인트: 다음 두 가지 키워드를 꼭 기억해둬.

트램펄린(Trampoline) 패턴: 재귀 함수가 직접 다음 재귀를 호출하지 않고, 함수를 반환(반환값을 함수로 감싸기)하게 해. 이를 루프 안에서 계속 실행하는 방식으로, 재귀 호출을 루프 안의 함수 실행으로 바꾸어 스택이 쌓이지 않게 하는 기법이야.
```javascript
// 1. 트램펄린 함수: 반환된 함수가 더 이상 함수가 아닐 때까지 루프를 돌림
const trampoline = (fn) => (...args) => {
  let result = fn(...args);
  while (typeof result === 'function') {
    result = result(); // 계속 실행!
  }
  return result;
};

// 2. 재귀를 함수 반환으로 대체 (Thunk)
const sumRecursive = (n, acc = 0) => {
  if (n <= 0) return acc;
  // 함수를 반환함으로써 호출 스택을 여기서 종료시킴
  return () => sumRecursive(n - 1, acc + n);
};

// 3. 실행
const wrappedSum = trampoline(sumRecursive);
console.log(wrappedSum(100000)); // 스택 오버플로우 없이 동작!
```

명시적 스택(Explicit Stack) 사용: 재귀를 호출할 때 사용하는 '콜 스택' 대신, 우리가 직접 Array나 Stack 자료구조를 만들어서 호출할 정보들을 담고 반복문으로 처리하는 방식이야. 즉, "시스템 스택을 사용하지 않고 내 힙(Heap) 메모리에 스택을 구현"하는 거지.

```javascript
function sumWithExplicitStack(n) {
  let stack = [];
  let sum = 0;
  
  // 1. 시스템 스택 대신 우리만의 스택에 시작 값 푸시
  stack.push(n);
  
  // 2. 루프를 돌며 직접 스택 관리
  while (stack.length > 0) {
    let current = stack.pop();
    
    if (current <= 0) continue;
    
    sum += current;
    // 3. 재귀 호출 대신 다음 값을 스택에 추가
    stack.push(current - 1);
  }
  
  return sum;
}

console.log(sumWithExplicitStack(100000)); // 힙 메모리까지만 사용 가능
```
