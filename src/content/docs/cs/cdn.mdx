### CDN이 등장한 이유

한국에서 해외 웹사이트에 접속했는데 이미지가 한 장씩 늦게 나타난 경험이 있을 것이다.

웹사이트 서버가 미국에 있다면 한국 사용자의 요청은 미국까지 이동해야 한다. 서버가 응답한 이미지와 JavaScript 파일도 다시 긴 네트워크 경로를 따라 한국으로 돌아와야 한다.

빛의 속도가 빠르다고 해도 데이터가 목적지까지 곧장 이동하는 것은 아니다. 여러 네트워크 장비와 통신 사업자를 거치며 이동하기 때문에 물리적인 거리가 멀수록 지연 시간도 커진다.

이 문제를 해결하기 위해 등장한 것이 CDN이다.

### CDN이란

CDN은 Content Delivery Network의 약자이다.<br/>
웹 사이트에 접속할 경우 인터넷을 통해 사이트의 데이터가 사용자 컴퓨터로 이동하는데
이 시간을 단축시키기 위해 CDN이 존재한다.<br/>
따라서 CDN은 웹 페이지 로드 속도를 높이는 콘텐츠 전송 네트워크이다.

콘텐츠 : 웹사이트의 이미지, CSS, JavaScript

사용자와 가까운 엣지 서버에 저장하고 전달

<pre>
본사 창고: 오리진 서버<br/>
지역 물류센터: CDN 엣지 서버<br/>
물류센터가 모여 있는 거점: PoP<br/>
저장된 상품: 캐시된 콘텐츠<br/>
가까운 물류센터에서 배송: 캐시 히트
</pre>

### CDN이 전송하는 콘텐츠

두가지 유형이 존재한다.

1. **정적 콘텐츠**<br/>
   이 콘텐츠는 사이트에서 주로 바뀌지 않는 부분이다.
   mdn으로 예시를 들면 헤더 이미지, 로고, 배너가 된다. 이는 바뀌지 않는 값으로 수정할 필요가 없기 때문에 CDN에 저장하는게 이상적이다.
2. **동적 콘텐츠**<br/>
   날씨, 로그인 상태 같이 자주 바뀔 수 있는 부분이다.
   또한 사용자별로 다르게 나타나야 하는 부분일 수 있다. 이것도 CDN을 통해 전송할 수 있는 콘텐츠 중에 하나다.

### CDN 동작 원리

CDN은 POP 또는 엣지 서버 그룹으로 작동된다.<br/>
캐싱, 동적 가속, 엣지 로직 원리를 기반으로 작동한다.

### CDN 캐싱?

#### Cache Miss
처음 user가 콘텐츠를 요청하면 가까운 edge 서버에 요청이 도착한다. <br/>
하지만 첫 요청이기때문에 캐싱된 데이터가 존재하지 않는다. 이를 Cache Miss 라고 한다.

#### Cache Hit
그 다음 요청부터는 캐싱되어있는 데이터를 기반으로 새로운 요청을 보내지 않아도 된다.<br/>
바로 근처의 edge 서버에서 콘텐츠를 반환한다. 이를 Cache Hit라고 한다.

#### 그러면 캐싱할 수 있는 콘텐츠의 영역은 어디까지일까?

변경되지 않는 콘텐츠는 캐싱하기 유용하다. -> static contents
하지만 user마다 달라질 수 있는 데이터는 캐싱하기 어렵다. -> dynamic contents

⚠️ 하지만 동적 컨텐츠 즉 날씨 같은 동적 컨텐츠는 캐싱 할 수 있다. 단기간에 여러 user에게 같은 정보를 줘야하기 때문에 ⚠️

#### 어떻게 캐싱된 콘텐츠를 찾을까?
- 요청 URL
- 쿼리 문자열
- 요청 메서드
- 특정 요청 헤더
- 쿠키
- 국가나 디바이스 정보
위의 정보들이 담긴 cache key를 이용해서 캐싱된 콘텐츠를 찾는다. 

참고로 캐싱된 데이터는 TTL가 존재한다.

``` Cache-Control: max-age=3600 ```


#### 캐싱하지 못하는 데이터를 빠르게 전달한다는 의미

**기존 데이터 이동 경로**
```
사용자
  ↓
통신사 네트워크
  ↓
여러 중간 라우터
  ↓
국제망
  ↓
다른 국가의 통신사
  ↓
오리진 서버
```
TCP 연결이나 TLS 연결 과정도 이 사이에 여러번 왔다갔다 하는 방식으로 이루어진다. <br/>
그리고 연결하는 과정에서 보안 연결도 항상 이루어져야한다. 

**cdn 방식**
```
사용자
  ↓
가까운 CDN 엣지
  ↓
CDN 네트워크
  ↓
오리진 서버
```
하지만 CDN은
```
사용자 A ─┐
사용자 B ─┼→ CDN 엣지 ═════ 기존 연결 재사용 ═════ 오리진
사용자 C ─┘
```

이런식으로 연결을 단순화 하고 캐싱하여 재사용할 수 있다. <br/>
위의 과정을 **동적 과정**이라고 한다.
이는 **지능형 라우팅 알고리즘**을 사용하는데 이에 관해서는 이후에 포스팅 하겠다.

### Edge Computing

엣지 컴퓨팅이란,<br/>
만약 오리진이 미국에 있고 요청한 국가가 한국이라고 가정을 해본다.

기존 로직은
```
한국 사용자
→ CDN
→ 미국 오리진 서버
→ 접속 국가 확인
→ /ko로 리다이렉트 응답
→ 사용자
```
이렇게 된다.<br/>
하지만 엣지 컴퓨팅을 이용한다면
```
한국 사용자
→ 가까운 CDN 엣지
→ 접속 국가 확인
→ /ko로 리다이렉트
```
위처럼 통신 시간을 단축할 수 있다.

엣지와 캐시가 좀 헷갈린다 뭔가 비슷해보이지만 아래의 표로 설명하면 이해가 쉽다.
| 구분     | 캐싱         | 엣지 컴퓨팅          |
| ------ | ---------- | --------------- |
| 핵심     | 저장된 응답 재사용 | 엣지에서 코드 실행      |
| 예시     | 이미지 반환     | 국가별 리다이렉트       |
| 오리진 요청 | 생략 가능      | 로직에 따라 생략 가능    |
| 결과     | 기존 응답 반환   | 판단하거나 가공한 결과 반환 |

#### "CDN에 정적 파일(JS, CSS, 이미지 등)을 캐싱해 두면 성능상 이점이 크지만, 만약 급하게 서비스 코드를 수정해서 배포했을 때 사용자가 여전히 옛날 캐시 파일을 보게 되는 문제가 생길 수 있습니다. 이를 해결하기 위해 실무에서는 보통 어떤 전략들을 사용하나요?"

```헤더에 캐시 유지 타임을 빼서 보냅니다.```

**객관적 점수: 50점 / 100점**<br/>
이유: 반은 맞고 반은 아쉽습니다. 헤더에서 캐시 유지 타임(max-age=0 또는 no-cache)을 빼거나 조절하는 것은 훌륭한 접근입니다. 다만, 이 방식은 "매번 오리진 서버에 물어봐야 하므로 CDN의 캐싱 이점을 포기해야 한다"는 치명적인 `트레이드오프`가 생깁니다. `빌드 도구를 활용한 프론트엔드 배포`의 정석적인 답변이 누락되었습니다.

"헤더의 캐시 타임을 조절하는 것도 방법이지만, 실무에서는 캐싱 효율을 극대화하기 위해 파일명에 해시값을 붙이는 Content Hashing 전략을 가장 많이 씁니다."

보안 답변 가이드:<br/>
"우선 index.html 같은 진입점 파일은 헤더 설정을 통해 캐싱하지 않거나(no-cache) TTL을 극도로 짧게 가져갑니다. 대신, 웹팩이나 비트(Vite) 같은 빌드 툴을 활용해 JS, CSS, 이미지 같은 정적 자원들은 파일 내용이 바뀔 때마다 main.a1b2c3.js처럼 파일명에 고유한 해시값이 붙도록 빌드합니다.
이렇게 하면 파일 이름이 달라지기 때문에 CDN 엣지와 브라우저는 이를 완전히 새로운 파일로 인식하여 자동으로 새 버전을 다운로드(Cache Miss)하게 됩니다. 만약 파일명을 바꿀 수 없는 상황에서 급한 수정이 필요하다면, CDN 인밸리데이션(Invalidation/Purge) 기능을 호출해 전 세계 엣지의 캐시를 강제로 비워주는 전략을 사용합니다."

#### "블로그에 '날씨 데이터 같은 동적 콘텐츠도 단기간 동안 여러 유저에게 동일하게 보여주기 위해 캐싱이 가능하다'고 잘 정리해 주셨는데요. 그렇다면 이 동적 데이터의 캐시 유효 기간(TTL)은 실무에서 보통 어떤 기준으로 설정해야 할까요? 만약 TTL을 너무 길게 주거나 너무 짧게 주면 각각 어떤 문제가 생길까요?"

```만약 TTL을 너무 길게 사용하면 동적으로 콘텐츠 자체가 변경되어야 할때 캐싱으로 인해서 내용이 바뀌지 않는 현상이 발생할 것입니다. 따라서 콘텐츠 유형에 따라서 TTL을 조율해야합니다. 예를 들어서 날씨 같은 콘텐츠는 날씨가 바뀌는 시간을 고려해서 캐싱을하거나 할 것 같습니다. 대신 너무 짧게 주게 되면 캐싱하는 의미가 없어집니다.```

**객관적 점수: 90점 / 100점**<br/>
이유: 출제 의도를 완벽하게 파악하셨습니다! TTL이 길 때의 부작용(데이터 불일치)과 짧을 때의 부작용(캐싱 의미 퇴색)을 정확하게 짚으셨고, 날씨 도메인을 엮어 비즈니스 요구사항에 따라 조율해야 한다는 논리도 아주 훌륭합니다.

💡 면접관 피드백 & 문장 업그레이드<br/>
"완벽합니다. 여기에 '오리진 서버 부하'라는 단어만 명시적으로 얹어주면 100점짜리 답변이 됩니다."

보안 답변 가이드:<br/>
"정확합니다. 동적 콘텐츠의 TTL은 비즈니스가 허용하는 데이터의 '실시간성'과 '서버 부하' 사이의 타협점을 찾아 설정해야 합니다. 슬님 말씀대로 TTL이 지나치게 길면 유저가 업데이트되지 않은 과거의 데이터를 보게 되어 서비스 신뢰도가 떨어집니다. 반면, TTL을 너무 짧게 주면 캐시가 수시로 만료되어 결국 모든 트래픽이 오리진 서버(Origin Server)로 그대로 쏟아지기 때문에, CDN을 도입했음에도 오리진 서버의 CPU와 대역폭 비용이 급증하는 문제가 발생합니다. 따라서 데이터의 변화 주기(예: 날씨 데이터의 갱신 주기)를 고려해 최적의 TTL을 산정해야 합니다."

#### "블로그에 캐싱된 콘텐츠를 찾을 때 요청 URL이나 쿼리 문자열(Query String)을 담은 Cache Key를 이용한다고 정리하셨습니다. 만약 마케팅 부서에서 광고 추적용 파라미터(예: ?utm_source=facebook)를 URL 뒤에 붙여서 유저들을 대거 유입시켰을 때, CDN 단에서 어떤 비효율이 발생할 수 있고 이를 어떻게 방지할 수 있을까요?"

```cdn에 과부화가 올 것 같습니다. key를 이용해서 찾는 방법은 해시맵 알고리즘과 같은 방식으로 되어있을텐데 이는 메모리 점유율이 크기 때문에 대거 유입이 된다면 과부화가 올거라고 생각합니다.```

**객관적 점수: 35점 / 100점**<br/>
이유: 컴퓨터 과학(CS) 관점에서 해시맵 구조와 메모리 점유율을 떠올리신 접근은 매우 창의적이고 좋았습니다! 하지만 실제 인프라 환경에서 글로벌 CDN 기업(Cloudflare, AWS CloudFront)의 대규모 분산 서버들은 고작 쿼리 스트링 수만 개 추가된다고 해서 메모리 과부하가 오지 않습니다. 진짜 문제는 CDN 서버가 아니라, "CDN이 제 역할을 못 해서 엉뚱한 오리진 서버(우리 본사 서버)가 폭파된다"는 점입니다.

💡 면접관 피드백 & 문장 업그레이드<br/>
"CDN 자체의 과부하보다는, 수많은 캐시 미스(Cache Miss)가 발생해 뒤에 있는 '오리진 서버'가 트래픽 폭탄을 맞게 된다는 점을 짚어야 합니다."

보안 답변 가이드:<br/>
"CDN 서버 자체의 메모리 과부하보다는, 우리 오리진 서버의 과부하가 발생합니다. CDN은 기본적으로 전체 URL(쿼리 스트링 포함)을 기반으로 Cache Key를 만드는데, 마케팅 파라미터(?utm_source=facebook, ?utm_source=google)가 붙으면 내부 콘텐츠(HTML, 이미지)가 완전히 똑같음에도 불구하고 CDN은 전부 다 다른 페이지로 인식합니다.
이로 인해 유저들이 대거 유입될 때 CDN 엣지에서 캐시 미스(Cache Miss)가 대량으로 발생하게 되고, 엣지가 이 모든 요청을 우리 오리진 서버로 그대로 토스하게 되면서 오리진 서버가 트래픽을 버티지 못하고 다운되는 비효율이 발생합니다.
이를 방지하기 위해 실무에서는 CDN 설정에서 페이지 렌더링에 영향을 주지 않는 광고 추적용 파라미터(utm_ 등)를 Cache Key 계산에서 제외(Ignore/Strip Query Strings)하도록 룰을 세팅하거나, 엣지 컴퓨팅을 활용해 엣지 단에서 해당 쿼리를 제거한 후 캐시를 조회하도록 아키텍처를 구성합니다."
