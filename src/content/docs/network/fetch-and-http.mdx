## 🫨 fetch API란?

fetch의 사전적 의미 : (무엇을) 가져오다, (무엇을) 얻다 <br/>
브라우저에 기본으로 내장된 자바스크립트 전역 메서드(함수)

과거에는 XMLHttpRequest를 사용하거나 AJAX나 axios같은 무거운 라이브러리를 사용해야했다.<br/>
하지만 최근에는 웹브라우저에 표준으로 fetch가 내장되어 네트워크 통신이 간편해졌다.<br/>
(2017~2018년부터 대부분의 브라우저)

## 😎 fetch의 특징

- Promise 기반이다.

```javascript
// fetch에게 리소스 경로 전달
fetch("https://api.example.com/data")
  .then((response) => response.json())
  // 서버가 request header를 들고 오면 response 객체가 생성된다.
  // 이후 json으로 파싱
  .then((data) => console.log(data))
  .catch((error) => console.error("Error:", error));
```

- 404나 500 같은 에러가 발생해도 reponse를 뱉었다고 간주하기 때문에 catch로 잡히지 않는다. - Resolve로 처리한다.<br/>
  따라서 response.ok 나 response.status로 에러를 체크해야한다.

### 😯 여기서 Worker란?

Window 컨텍스트: DOM을 조작하는 메인 화면 환경<br/>
Worker 컨텍스트: 무거운 연산이나 데이터 처리를 전담하는 백그라운드 환경<br/>

브라우저는 기본적으로 싱글스레드이다.<br/>
DOM을 조작하는데에 바쁜 Window 컨텍스트에서 무거운 연산이 발생하면 UI가 멈추는 현상이 발생할 수 있다.

fetch()는 Window와 Worker 컨텍스트 양쪽에서 모두 사용할 수 있는 전역 메서드라는 점을 고려한다면,<br/>
Worker 컨텍스트를 사용해서 백그라운드에서 네트워크 요청을 보낼 수 있다.

```javascript
// [Window 컨텍스트 - 메인 스레드 script.js]

async function loadData() {
  try {
    // 1. 메인 스레드가 직접 fetch 호출
    const response = await fetch("https://api.example.com/heavy-data");
    if (!response.ok) throw new Error("Network error");
    const data = await response.json();

    // 2. 메인 화면(DOM)을 직접 만지거나 UI 상태를 바로 업데이트
    document.getElementById("result").innerText = data.message;
  } catch (error) {
    console.error(error);
  }
}
```

```javascript
// [Window 컨텍스트 - script.js]

// 1. 백그라운드 Worker 생성
const myWorker = new Worker("worker.js");

// 2. fetch 요청을 Worker에게 전달
myWorker.postMessage({
  action: "FETCH_DATA",
  url: "https://api.example.com/heavy-data",
});

// 3. Worker가 fetch 결과물을 메인 스레드로 보내면 받는 이벤트 핸들러
myWorker.onmessage = function (event) {
  const data = event.data;
  // 4. 받은 결과물로 메인 화면 UI 업데이트 -> window 컨텍스트에서 DOM 조작
  document.getElementById("result").innerText = data.message;
};

// [Worker 컨텍스트 - worker.js]

// window가 요청하면 실행하는 이벤트 핸들러
self.onmessage = async function (event) {
  if (event.data.action === "FETCH_DATA") {
    try {
      const response = await fetch(event.data.url);
      const data = await response.json();

      // 작업이 다 끝나면 메인 스레드(Window)에 전달
      self.postMessage(data);
    } catch (error) {
      self.postMessage({ error: error.message });
    }
  }
};
```

## AI 질문

### Q1. "블로그에 fetch는 404나 500 에러 코드를 받아도 catch 블록으로 가지 않고 Resolve(이행)된다고 정리해 주셨습니다. fetch API 명세가 설계될 때, 왜 에러 상황에서도 거부(Reject)하지 않고 이행하도록 메커니즘을 설계했을까요?"

404나 500 에러 또한 서버의 응답으로 간주하여 통신이 되었다면 Resolve로 처리하고 통신이 끊긴것이 아닌 에러는 코드 단계에서 개발자들이 자류롭게 설계할 수 있도록 하기 위해서 입니다.

#### ai 피드백

- '통신이 끊겼다'는 표현을 "네트워크 레이어 레벨의 실패" 로 변환
- fetch API 스펙상 통신의 실패(Reject) 기준을 'HTTP 프로토콜의 성립 여부'로 규정했기 때문..이라고 설명

### Q2. "Window와 Worker 컨텍스트 모두에서 fetch를 쓸 수 있고, 작성해 주신 예시를 보면 Worker 환경을 쓰면 메인 스레드를 방해하지 않는 장점이 있다고 하셨습니다. 그렇다면 모든 fetch 요청을 Web Worker로 넘겨서 처리하면 항상 이득일 것 같은데, 실제 서비스의 모든 API 요청을 Worker로 이관하지 않는(즉, Window에서 그냥 fetch를 쓰는) 이유는 무엇인가요? 구조적인 단점이나 비용 측면에서 설명해 주세요."

쓰레드를 여러개 사용하는 것은 비용 측면에서 반드시 옳다고 할 수 없습니다. 무거운 연산이 포함된 fetch 요청이 아니라면 즉시 main thread에서 DOM에 빠르게 반영하는 것이 더 효율적일 수 있기 때문입니다. 만약 간단한 연산에 의해 즉각 DOM의 변화가 필요한 상황이라면 window 객체에서 직접 fetch 요청을 실행 할수도 있다고 생각합니다.

#### ai 피드백

- postMessage 통신 오버헤드(비용)와 메모리 격리 구조에 대한 언급
- Window와 Worker는 메모리 공간이 완전히 격리되어 있어 데이터를 주고받을 때 postMessage API를 통한 '구조화된 복제(Structured Clone)' 프로세스가 일어납니다. 이 과정에서 데이터의 직렬화/역직렬화 및 메모리 복사 비용이 발생하므로, 무거운 연산이 없는 가벼운 fetch 요청까지 모두 Worker로 넘기면 오히려 이 통신 비용 때문에 전체적인 레이턴시가 늘어날 수 있습니다.

## 🫨 웹(Web)을 지탱하는 4대 발명품

- HTML (그릇): 정보를 하이퍼텍스트(링크) 형태로 담아두는 문서 형식.
- HTTP (규격): HTML을 컴퓨터끼리 오차 없이 주고받기 위해 만든 통신 규약.
- WorldWideWeb (클라이언트): 문서를 화면에 그리고 사용자에게 보여주는 최초의 브라우저.
- httpd (서버): 누군가 요청하면 하이퍼텍스트 문서를 꺼내서 던져주는 최초의 웹 서버 프로그램.

### HTTP/0.9 - 원라인 프로토콜

```
// 클라이언트의 요청
GET /mypage.html
```

```
// 서버의 응답
<html>
  A very simple HTML page
</html>
```

한줄만 주고 받아서 원라인 프로토콜

#### 특징

- header가 없으니까 content-type, content-length 같은 정보가 없다.
- states code도 없다. (200 OK, 404 Not Found 같은 코드가 없다.)
- get 요청만 지원한다. (post, put, delete 같은 다른 메서드가 없다.)

### HTTP/1.0

- 버전 정보를 포함 (HTTP/1.0)
- HTTP 헤더 도입 (Content-Type, Content-Length, User-Agent 등)
- 상태 코드 도입 (200 OK, 404 Not Found 등)

⚠️ 한계

- 연결마다 TCP 3-way Handshake를 해야해서 연결 설정이 느리다. (Connection Overhead)

### HTTP/1.1

- keep-alive 연결 도입 (연결 재사용)
- 파이프라이닝 지원 (여러 요청을 한 번에 보내고 응답을 기다리지 않고 처리)
- Rest API의 등장 (HTTP 메서드: GET, POST, PUT, DELETE 등)
- CORS (Cross-Origin Resource Sharing) 도입

⚠️ 한계
Application HOLB

- 요청은 연속적으로 보낼 수 있지만 응답은 순차적으로 돌아온다. (Head of Line Blocking)

### HTTP/2

- 구글의 SPDY 프로토콜을 기반
- 텍스트 형식의 HTTP/1.x에서 바이너리 형식으로 변경 (Binary Protocol)
- 멀티플렉싱 지원 (한 연결에서 여러 요청과 응답을 동시에 처리)
- 헤더 압축 (Header Compression) -> 중복 헤더 압축

⚠️ 한계
Transport(TCP) HOLB
TCP는 데이터 유실이 나면 무조건 멈춰서 재전송을 기다림 <br/>
즉 패킷 하나가 유실되면 유실되지 않은 다른 스트림까지 전송 계층에서 멈춰버리는 현상

### HTTP/3

- 멀티플렉싱을 지원하되, 각 요청을 독립된 스트림으로 처리하는 QUIC 프로토콜 기반
- TCP처럼 연결할 때 여러 번 Handshake하지 않아서 연결 설정 속도(0-RTT)가 극도로 빠르다.

### Q1. "블로그에 HTTP/1.1의 한계로 Application HOLB를 적어주셨고, 이를 해결하기 위해 HTTP/2에서 멀티플렉싱이 도입되었다고 하셨습니다. 그렇다면 과거 HTTP/1.1 시절에 우리 선배 개발자들이 이 HOLB 병목을 우회하고 로딩 속도를 높이기 위해 유저단(프론트엔드 빌드 시점 등)에서 눈물겹게 썼던 최적화 기법들이 무엇이 있을지 아는 대로 설명해 주세요."

통신 과정에서 문제가 발생하면 Resolve되지 않기 때문에 이 경우를 판별하여 재요청을 보내는 기법을 사용했을 것 같습니다.

### Q2. "HTTP/2의 멀티플렉싱 덕분에 단 하나의 TCP 연결로 모든 리소스를 주고받을 수 있게 되었는데, 역설적이게도 이 때문에 Transport(TCP) HOLB라는 새로운 문제가 터졌다고 정리해 주셨습니다. 왜 단 하나의 커넥션만 사용하는 구조가 패킷 유실 상황에서 HTTP/1.1보다 더 치명적인 문제가 될 수 있는지, 그리고 HTTP/3는 UDP를 통해 이를 어떻게 극복했는지 한 단계 더 깊게 설명해 주실 수 있나요?"

단하나의 커넥션이 위험 한 것은 터널을 예시로 생각하면 이유가 명확합니다. 아무리 많은 짐을 효율적으로 담은 트럭들을 보낸다고 한들 하나의 터널에서 앞에 길을 막아버리면 뒤의 방대한 짐을 가지고 있는 트럭이 움직이지 못하고 도착지로 도달하지 못하기 때문입니다. HTTP/1.1은 요청마다 커넥션을 만들기 때문에 연결의 끊김을 판별하고 처리하기에 HTTP/2보다 유연한 구조라고 볼 수 있습니다. HTTP/3는 이 문제를 해결하기 위해 아예 터널을 여러개 뚫어버린 상황입니다. 따라서 다른 통신들에 영향을 받지 않기 때문에 이를 완벽하게 극복했다고 볼 수 있습니다.

#### ai 피드백

최종 점수: 95점 <br/>
표준/표현력 (90점): 비유는 완벽했으니, 실제 면접장에서는 터널을 여러 개 뚫었다는 표현 뒤에 "UDP 기반의 독립적인 스트림(Stream) 구조"라는 기술 명칭만 싹 얹어주면 완벽한 100점짜리 무기가 될 거야.

### Q3. "작성해주신 블로그를 보면 HTTP/3는 TCP의 Handshake 과정을 생략하여 0-RTT 수준의 극도로 빠른 연결 설정 속도를 제공한다고 하셨습니다. 프론트엔드 개발자 입장에서, 이 HTTP/3의 '0-RTT'라는 특성이 글로벌 유저를 타겟으로 하거나 네트워크 환경이 불안정한 모바일 웹 앱 서비스를 운영할 때 어떤 사용자 경험(UX)적 이점을 가져다줄 수 있을까요?"

어떤 위치에 있던 빠르게 연결되고 데이터 통신이 가능하기 때문에 속도 측면에서 압도적인 사용자 경험을 제공할 수 있습니다.

#### ai 피드백

최종 점수: 80점 <br/>
표준/표현력 (70점): "속도 측면에서 압도적"이라는 표현은 조금 추상적이야. 경력직 개발자라면 "모바일 기기에서 네트워크(와이파이 ↔ 셀룰러)가 전환되는 상황"이나 "FCP(최초 콘텐츠 렌더링 시간) 단축" 같은 실무적인 상황과 성능 지표를 엮어서 구체적으로 말해주는 게 점수를 따기 훨씬 좋아.
