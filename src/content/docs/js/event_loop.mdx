### event loop
JavaScript는 기본적으로 싱글 스레드 언어다.
<pre>JavaScript 엔진은 한 번에 하나의 작업만 실행할 수 있다.</pre>

그런데 브라우저에서 동작하는 화면을 보면 애니메이션이 실행되는 동안 네트워크 요청도 처리되고 사용자의 클릭 이벤트도 동시에 반응하는 것처럼 보인다.

이것이 가능한 이유는 브라우저가 제공하는 Web API와 이벤트 루프가 함께 동작하기 때문이다.

`setTimeout, DOM 이벤트, 네트워크 요청` 같은 작업은 JavaScript 엔진이 직접 계속 붙잡고 처리하지 않는다.<br/>
이런 작업들은 브라우저의 Web API 영역으로 전달되고 JavaScript 엔진은 콜스택에 있는 다음 작업을 계속 실행한다.

Web API에서 작업이 완료되면 그 작업의 콜백 함수는 바로 실행되지 않고 task queue에 들어간다.

이벤트 루프는 콜스택과 task queue를 계속 확인한다.

콜스택이 비어 있고 task queue에 대기 중인 작업이 있다면<br/>
이벤트 루프는 task queue에서 가장 먼저 들어온 작업을 꺼내 콜스택으로 이동시킨다.<br/>
그때 콜스택에 올라간 콜백 함수가 실행된다.

이벤트 루프는 `콜스택이 비었을 때 대기 중인 작업을 실행할 수 있도록 순서를 조율하는 역할`을 한다.

이 구조 덕분에 JavaScript는 싱글 스레드로 동작하면서도 브라우저 환경에서 비동기 작업을 처리할 수 있다.

### event loop & browser

JavaScript 엔진은 **코드를 평가하고 실행하면서 실행 컨텍스트를 생성**한다.<br/>
생성된 후에는 실행 컨텍스트 스택에 push되고 실행이 끝나면 pop된다.

이 `실행 컨텍스트 스택`이 흔히 말하는 `콜스택`이다.

JavaScript는 **싱글 스레드**로 동작하기 때문에 콜스택이 하나다.<br/>
그래서 하나의 동작만 할 수 있다.

<i>반면 힙은 메모리 공간이다.</i>

JavaScript 엔진은 코드의 평가와 실행을 담당하고,<br/>
타이머, 네트워크 요청, DOM 이벤트 감지 작업은 `브라우저`나 `Node.js 런타임`이 담당한다.

예를 들어 setTimeout 코드를 만나면 JavaScript 엔진은 setTimeout 호출한다.<br/>
하지만 지정된 시간을 직접 세거나 콜백 함수를 바로 실행하지는 않는다.

타이머 설정과 콜백 등록은 브라우저 또는 Node.js 런타임이 맡아서 처리한다.<br/>
지정된 시간이 지나면 해당 콜백 함수는 바로 콜스택으로 들어가는 것이 아니라 task queue에 들어간다.

이후 이벤트 루프는 콜스택이 비었는지 계속 확인한다.<br/>
콜스택이 비어 있고 task queue에 대기 중인 콜백이 있다면 이벤트 루프는 가장 먼저 들어온 콜백을 콜스택으로 이동시킨다.

그때 콜스택에 올라간 콜백 함수가 JavaScript 엔진에 의해 실행된다.

### Task Queue

Task Queue는 비동기 작업이 완료된 뒤 실행되어야 할 콜백 함수가 일시적으로 대기하는 공간이다.<br/>
(정확히는 Macrotask Queue라고도 부른다. 이는 Promise가 대기하는 Microtask Queue와 다르다.)

콜스택은 현재 실행 중인 함수들이 쌓이는 곳<br/>
Task Queue는 비동기 작업이 완료된 후 실행을 기다리는 콜백 함수들이 대기하는 곳이다.

### Event Loop와 Task Queue 실행 예시

```javascript
function foo() {
  console.log('foo');
}

function bar() {
  console.log('bar');
}

setTimeout(foo, 0);
bar();
```

1. 전역 코드가 평가되고 전역 실행 컨텍스트가 생성되어 콜스택에 push된다.
2. 전역 코드가 실행되면서 `setTimeout(foo, 0)`이 호출된다.
3. `setTimeout` 실행 컨텍스트가 `콜스택에 push`된다.
4. `setTimeout`은 브라우저의 타이머 기능에게 `foo` 콜백을 일정 시간 이후 실행하도록 맡긴다.
5. `setTimeout` 실행이 끝나고 콜스택에서 pop된다.
6. 이어서 `bar()`가 호출되고 `bar` 실행 컨텍스트가 콜스택에 push된다.
7. `bar` 함수 안의 `console.log('bar')`가 실행된다.
8. `bar` 실행이 끝나고 콜스택에서 pop된다.
9. 전역 코드 실행이 끝나고 전역 실행 컨텍스트가 콜스택에서 pop된다.
10. 타이머가 만료되면 브라우저는 콜백 함수 foo를 Task Queue에 push한다.<br/>
(이 과정은 6~9번 과정이 메인 스레드에서 일어나는 동안 백그라운드에서 병렬로 진행된다.)
11. 이벤트 루프는 콜스택이 비어 있는지 확인한다.
12. 콜스택이 비어 있으면 Task Queue에 있던 `foo`를 콜스택으로 이동시킨다.
13. `foo`가 실행되면서 `console.log('foo')`가 출력된다.

따라서 출력 순서는 다음과 같다.

```javascript
bar
foo
```

`setTimeout(foo, 0)`은 `foo`를 즉시 실행한다는 뜻이 아니다. <br/>
브라우저에게 타이머를 맡기고 타이머가 만료되면 `foo` 콜백이 Task Queue에 들어간다. <br/>
이후 이벤트 루프가 콜스택이 비어 있는 것을 확인했을 때 `foo`가 실행된다.

## Task Queue의 이원화 (Macro vs Micro)

위에서 살펴본 Task Queue는 사실 **매크로태스크 큐(Macrotask Queue)**다.<br/>
이벤트 루프는 비동기 작업을 모두 똑같이 처리하지 않고 명확한 우선순위가 존재한다.

### 1. Microtask Queue (우선순위: 최상위)
* **종류:** `Promise.then` `catch` `finally` `async` `await`
* **특징:** 콜스택이 비면 무조건 `Microtask Queue` 큐를 먼저 확인하며 큐가 완전히 비워질 때까지 쌓여있는 모든 작업을 다 처리한다.

### 2. Macrotask Queue (우선순위: 낮음)
* **종류:** `setTimeout`, `setInterval`, DOM 이벤트 등
* **특징:** 루프 한 번에 딱 '하나'의 태스크만 꺼내서 콜스택에 올린다.

<pre>
콜스택이 비어야 한다.
Microtask Queue가 비어야 한다.
그다음 Macrotask Queue에서 하나를 꺼낸다.
</pre>

```javascript
console.log('1. Start');

setTimeout(() => {
  console.log('2. Timeout (Macrotask)');
}, 0);

Promise.resolve()
  .then(() => {
    console.log('3. Promise 1 (Microtask)');
  })
  .then(() => {
    console.log('4. Promise 2 (Microtask)');
  });

console.log('5. End');
```

1. **동기 코드(Synchronous) 실행 주도**
   - 최상단 `console.log('1. Start')`가 콜스택에 쌓이고 즉시 실행되어 출력
   - `setTimeout`은 브라우저 Web API로 타이머 처리를 위임
   - `Promise.then` 콜백은 **Microtask Queue**에 등록
   - 최하단 `console.log('5. End')`가 콜스택에 쌓이고 즉시 실행되어 출력
   - 전역 실행 컨텍스트까지 끝나면서 **Call Stack이 완전히 비워짐**

2. **이벤트 루프의 Microtask Queue 우선 점검**
   - 콜스택이 비자마자 이벤트 루프는 가장 우선순위가 높은 Microtask Queue를 확인.
   - 큐에 있던 `3. Promise 1`이 실행되고 `4. Promise 2`까지 큐가 완전히 비워질 때까지 연속으로 콜스택에 올려 실행

3. **최종 Macrotask Queue 실행**
   - 마이크로태스크 큐까지 텅 빈 것을 확인 -> 대기 중이던 `2. Timeout` 콜백을 콜스택으로 이동시켜 실행
