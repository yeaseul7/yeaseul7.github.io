### scope 란?

값(변수)과 "표현식"이 현재 실행되는 컨텍스트를 의미한다. <br/>
만약 변수나 표현식이 해당 스코프 안에 있지 않다면 사용할 수 없다. <br/>
scope는 포함 구조를 가진다. <br/>
즉 상위 크로프가 하위 스코프를 포함함에 따라서 하위 -> 상위로는 접근이 가능하지만 상위 -> 하위로는 접근이 불가능하다.

### scope의 종류

- 전역 스코프 : 어디서든 접근 가능한 스코프
- 함수 스코프 : 함수 내부에서만 접근 가능한 스코프 -> function으로 생성된 범위
- 블록 스코프 : 블록 내부에서만 접근 가능한 스코프
- 모듈 스코프 : 모듈 내부에서만 접근 가능한 스코프

### scope와 hoisting의 관계

scope가 변수와 함수가 유효한 범위라면 <br/>
hoisting은 scope 안의 선언을 실행 전에 등록하는 과정이다.

#### 질문 1

```javascript
console.log(a);

var a = 10;
```

실행 결과는?<br />
...
undefined

이유는 javascript의 런타임 중 (한줄씩 실행되기 바로 전 단계)에 var를 보고 메모리를 할당하기 때문이다. <br/>
따라서 a는 선언이 호이스팅되어 undefined로 초기화된 상태로 존재

#### 질문 2

```javascript
function test() {
  console.log(name);

  var name = "철수";
}

test();
```

실행 결과는?<br />
...
undefined

이유는 name은 전역에 등록되는 게 아니라, test 함수 스코프 안에 등록된다. <br/>
실제로 동작하는 코드는 아래와 같다.

```javascript
function test() {
  var name;

  console.log(name);

  name = "철수";
}
```

### const와 let은 다르다.

호이스팅은 되지만, 미리 등록은 되지만 선언문을 만나기 전에 접근할 수 없는 TDZ(Temporal Dead Zone)에 빠진다. <br/>

```javascript
{
  // age는 이 블록 스코프에 등록되어 있지만 아직 접근 불가
  console.log(age); // ReferenceError

  let age = 20;
}
```

### 함수 선언문도 호이스팅된다.

함수 선언문은 실행 전에 해당 스코프에 함수 자체가 등록되기 때문이다.

```javascript
hello();

function hello() {
  console.log("hi");
}
```

#### ⚠️ 함수 표현식은 호이스팅되지 않는다.

```javascript
hello();

var hello = function () {
  console.log("hi");
};
```

이 코드는 곧

```javascript
var hello;

hello(); // 이 시점에 hello는 undefined

hello = function () {
  console.log("hi");
};
```

### javascript는 코드를 실행하는 컴파일 과정에서 토크나이징과 렉싱이 일어난다. <br/>

토크나이징 : 코드가 토큰이라는 작은 단위로 나뉘는 과정 <br/>

```javascript
const name = "철수";
```

위 코드는 아래와 같이 토큰으로 나뉜다. <br/>

- const
- name
- =
- "철수"
- ;

렉싱 : 토큰이 어떤 종류인지 식별하는 과정 <br/>

| 코드 조각 | 토큰 종류     |
| --------- | ------------- |
| `const`   | 키워드        |
| `name`    | 식별자        |
| `=`       | 연산자        |
| `"철수"`  | 문자열 리터럴 |
| `;`       | 구분자        |

### 😯 무슨 관련이?

개발자가 코드를 작성할때 변수를 어디에 작성하는가를 바탕으로 따라서 렉시타임(렉싱)에 토큰이 분석되며 스코프가 결정되는 것

lexical은 쉽게 말하면 코드의 문자 구조, 코드가 작성된 위치와 형태

#### 질문 1. inner()는 어떤 x를 출력할까?

```javascript
const x = 1;

function outer() {
  const x = 10;

  function inner() {
    console.log(x);
  }

  inner();
}

outer();
```

답변 : 10;

이유 : inner 함수가 outer 함수 안에 있기 때문에 inner 함수의 렉시컬 환경에는 outer 함수의 렉시컬 환경이 포함되어 있다. <br/>
즉, inner 입장에서 바깥스코프는 x=1이 있는 전역이 아니라 outer 안에 있는 x=10이다.

#### 질문 2. inner()는 어떤 x를 출력할까?

```javascript
const x = 1;

function inner() {
  console.log(x);
}

function outer() {
  const x = 10;
  inner();
}

outer();
```

답변 : 1;
이유 : inner 함수가 정의된 위치가 전역이기 때문에 inner 함수의 렉시컬 환경에는 전역 스코프가 포함되어 있다. <br/>
즉, inner 입장에서 바깥스코프는 x=10이 있는 outer가 아니라 전역에 있는 x=1이다.

### 함수 스코프 var과 블록스코프가 비동기 함수를 만났을때 일어나는 일

```javascript
for (var i = 0; i < 5; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
```

위의 결과는 `5 5 5 5 5`이다. <br/>

var는 블록 스코프가 아니고 함수 스코프다. 이 의미는 곧

```javascript
var i;

for (i = 0; i < 5; i++) {
  setTimeout(function () {
    console.log(i);
  }, 1000);
}
```

var는 for안에 안묶인다. 함수 스코프 이기 때문에 <br/>
setTimout이 1초 뒤에 실행되는 시점에는 이미 i가 5가 되어버린 상태이다. 따라서 5가 5번 찍히는 것이다.

반면에 let은 어떨까?
let은 블록 스코프이다. 즉, for문 안에 묶인다. <br/>

```javascript
for (let i = 0; i < 5; i++) {
  setTimeout(function () {
    console.log(i);
  }, 1000);
}
```

따라서 for안에서 반복될때마다 새로운 렉시컬 환경을 만든다. <br/>
1번째 반복의 i = 0
└── 첫 번째 콜백이 이 i를 기억

2번째 반복의 i = 1
└── 두 번째 콜백이 이 i를 기억

3번째 반복의 i = 2
└── 세 번째 콜백이 이 i를 기억

## AI 면접 질문

### 1. "작성하신 글에서 let은 호이스팅되지만 초기화 전에 접근하면 ReferenceError가 발생한다고 하셨습니다. 그렇다면 자바스크립트 엔진은 let으로 선언된 변수를 호이스팅하는 것일까요, 아니면 하지 않는 것일까요? 만약 호이스팅한다면, 왜 선언문 이전에 접근했을 때 에러를 발생시키는 구조(TDZ)를 가지게 되었는지 설계적 관점에서 설명해주세요."

내 답변 <br/>
let도 호이스팅 과정을 거칩니다. let은 호이스팅으로 인해 먼저 메모리에 등록은 되지만 선언문까지 가는 동안에 공백이 발생합니다. 이걸 TDZ라고 하는데 let은 블록 스코프의 특징을 가짐으로 해당 스코프 안에서만 유효하다는 장점이 존재합니다. 이 TDZ가 존재하지 않는다면 이 특색이 무색해질 것입니다.

1번 질문: 호이스팅과 TDZ <br/>
평가: 85점

피드백: "공백이 발생한다"는 표현보다는 "선언 단계와 초기화 단계가 분리되어 있다"는 표현이 더 전문적이야. TDZ를 설계한 이유는 '런타임 에러를 사전에 방지'하기 위해서야. var처럼 선언 전에 사용해서 undefined라는 엉뚱한 값을 참조하게 두는 것보다, 개발자에게 명확한 에러를 던져서 코드 결함을 즉시 잡게 하려는 의도가 큼을 언급하면 완벽해.

### 2. "함수 선언문은 호이스팅되어 코드 상단에서 바로 호출 가능하지만, 함수 표현식은 var를 사용하더라도 호이스팅되지 않는다고 하셨습니다. 실행 컨텍스트의 '렉시컬 환경(Lexical Environment)' 생성 과정에서, 함수 선언문과 var로 선언된 함수 표현식이 엔진 내부에서 각각 어떤 단계(생성/초기화/할당)를 거쳐 준비되는지 그 차이를 상세히 설명해주세요."

내 답변 <br/>

먼저 코드를 토큰화하면 함수 표현식은 빈 변수를 먼저 선언하고 후에 함수를 집어 넣는 방식입니다. var를 사용하더라도 함수 자체가 호이스팅 되는것이 아니고 함수를 넣을 빈 변수만 호이스팅됩니다. 따라서 표현식의 렉시컬 환경은 빈 변수만 생성하고 초기화 이후에 함수를 할당하기 때문에 바로 호출이 불가능합니다

하지만 함수 선언문은 함수 자체를 먼저 생성하고 초기화 하기 때문에 코드 상단에서 바로 호출 가능합니다.

2번 질문: 함수 선언식 vs 표현식 <br/>
평가: 95점

피드백: 아주 정확해! 선언문은 평가 단계에서 환경 레코드에 함수 객체까지 생성해서 등록해버리기 때문에 즉시 호출 가능한 거야. 반면 표현식은 변수 식별자만 등록되고, 실제 함수가 할당되는 것은 런타임(코드 실행 단계)에 이루어지므로, 그전까지는 변수 이름만 알고 있을 뿐 함수라는 사실을 모르는 상태라는 점을 강조하면 최고야.

### 3. "작성하신 inner() 예제에서 inner 함수가 outer의 x를 참조하여 값을 출력하는 것은 클로저 현상입니다. 여기서 의문입니다. outer 함수가 호출을 마치고 실행 컨텍스트가 종료되면, outer의 렉시컬 환경도 소멸해야 하는 것이 원칙 아닌가요? 자바스크립트 엔진은 inner 함수가 존재하는 한 outer의 렉시컬 환경을 어떻게 보호(유지)하며, 이는 메모리 관리 측면에서 어떤 주의점을 시사하나요?"

내 답변 <br/>
클로저는 x의 값을 복사하는것이 아닌 대상을 가지고 있습니다. 즉, 계속해서 참조하고 있다는 의미입니다. inner함수가 outer의 x를 계속 참조하고 있다면 실행컨텍스트가 종료되어도 GC가 수집하지 않기 때문에 메모리 관리 측면에서 참조를 끊어 줄 필요가 있습니다.

3번 질문: 클로저와 메모리 관리<br/>
평가: 90점

피드백: 핵심인 '참조 유지'를 잘 설명했어. 엔진이 outer의 렉시컬 환경을 유지하는 이유를 더 기술적으로 풀자면, inner 함수의 [[Environment]] 내부 슬롯이 outer의 렉시컬 환경을 참조하고 있기 때문이야. GC(가비지 컬렉터)는 "어떤 스코프에서 이 변수를 참조 중인가?"를 보는데, inner가 살아있으니 outer의 환경도 계속 참조되는(Reachable) 상태로 남는 거지.

주의점: 무조건 참조를 끊는 게 답은 아니야. 오히려 클로저를 활용해 '데이터를 은닉'하고 '상태를 보존'하는 것은 아주 의도된 프로그래밍 패턴이니까, "필요 없는 경우에만 클로저를 의도치 않게 남발하지 않도록 주의한다"라고 덧붙이면 훨씬 성숙한 답변이 돼.
