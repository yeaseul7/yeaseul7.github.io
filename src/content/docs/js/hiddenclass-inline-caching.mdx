## javascript는 유연하고 동적인 언어이다.

javascript는 객체에 속성을 추가하고 삭제할 수 있다.<br/>
객체는 key-value 구조라서 해시 맵처럼 key로 value를 찾는 구조라고 생각할 수 있다. <br/>
javascript는 실제로 객체 속성 접근이 엄청 잦다.<br/>
이때마다 해시 처리한다면 비용이 매우 클 것이다.

java는 `Fixed Offset`을 사용해서 즉시 값에 접근한다. <br/>
Java에서 객체는 클래스에 의해 필드 구조가 고정되어 있다.<br/>
JVM은 각 필드가 객체 메모리 안에서 몇 바이트 떨어진 위치에 있는지 미리 알 수 있다.<br/>
-> 정해진 offset 위치를 기준으로 접근할 수 있다.

javascript는 Hidden Class / Shape 개념을 도입해서 위의 한계를 극복했다.

## hidden Class

```
C0: 빈 객체 상태
{}

C1: x가 추가된 상태
{ x }

C2: x와 y가 추가된 상태
{ x, y }
```
```javascript
const point = {
  x: 10,
  y: 20,
};
```
이렇게 있다고 하면
<pre>
point 객체
- hidden class: C2
- property storage: [10, 20]

C2
x -> offset 0
y -> offset 1
</pre>

위의 말은 곧<br/>
"x라는 속성은 몇 번째 칸에 있다"<br/>
"y라는 속성은 몇 번째 칸에 있다"<br/>
이것을 의미한다. 

#### Transition === 객체 모양의 변경 이력

속성이 추가될 때마다 객체의 구조 C0, C1, C2...이런식으로 변한다.<br/>
엔진은 그 구조에 맞는 새로운 히든 클래스로 이동시킨다.

위의 이력을 가지고 있기때문에<br/>
엔진은 객체가 어떤 순서로 속성을 추가했는지도 추적할 수 있다.

```javascript
const p1 = {};
p1.x = 10;
p1.y = 20;

const p2 = {};
p2.x = 30;
p2.y = 40;
```
위와 같이 있으면 `p1`과 `p2`는 같은 히든 클래스를 공유할 수 있다.

<pre>
`point.x`
→ point의 hidden class 확인
→ C2라면 x는 0번 칸
→ storage[0] 읽기
→ 10
</pre>

## Inline Casching

```javascript
function getX(point) {
  return point.x;
}

getX({ x: 10, y: 20 });
getX({ x: 30, y: 40 });
getX({ x: 50, y: 60 });
```

위와 같은 코드가 있을떄, `{ x, y }` 구조 객체들이 반복적으로 생성됐다.

#### `point.x`를 return할때 엔진이 하는 생각

<pre>
이전에 들어온 객체의 hidden class는 C2였다.
C2에서 x는 offset 0이었다.
다음에도 C2 객체가 오면 바로 offset 0을 읽으면 된다.
</pre>

이렇게 기억하고 실행하는게 `Inline Casching`이다.<br/>

만약 속성 추가 순서가 다르다면 결과가 달라질 수 있다. <br/>
위에서는 `p1`과 `p2`에 `x`와 `y`를 순서대로 넣었다. <br/>
하지만 반대로 넣는다면

```javascript
const p1 = {};
p1.x = 10;
p1.y = 20;

const p2 = {};
p2.y = 20;
p2.x = 10;

p1:
C0 -> add x -> C1 -> add y -> C2

p2:
C0 -> add y -> C3 -> add x -> C4
```
위처럼되면
```javascript
function getX(point) {
  return point.x;
}

getX(p1);
getX(p2);
```
어떤 객체는 C2고, 어떤 객체는 C4네?<br/>
C2에서는 x가 offset 0인데,<br/>
C4에서는 x가 offset 1일 수 있네?<br/>
이렇게 된다. 

이렇게 되면 Inline Cache도 단순해 지지않는다. 

그래서 객체를 일관된 방식으로 만드는 게 좋다.

따라서 `delete`도 안전하지 않다. <br/>
특정 객체의 구조가 안정적이지 않게 되기 때문에 해당 객체를 해시맵에 가까운 방식으로 바꿀 수 있다. <br/>
그러면 최적화에 어렵다.

## 심화 질문에 대답하기

### 구조적 변경과 최적화의 경계
**"만약 우리가 객체에 속성을 추가할 때 delete 연산자를 사용하여 중간에 특정 속성을 삭제하면, 엔진 내부의 히든 클래스 체인과 인라인 캐싱은 어떻게 반응할까요? 단순히 '성능이 떨어진다'를 넘어, 엔진이 해당 객체를 어떤 상태로 다시 취급하게 되는지 설명해 줄 수 있나요?"**

히든 클래스 체인은 비슷한 형식을 기반으로 빠르게 탐색하기 위한 최적화 방법인데 delete로 특정 객체의 형식을 갑자기 바꾼다면 엔진이 형식이 일관된다고 파악하지 않아서 해시맵 형태로 다시 바꾸게 됩니다. <br/>
그렇게 되면 delete 이전에 수많은 비슷한 형식을 가진 객체의 value를 찾을때 해시맵 형태로 찾기 때문에 key가 어딨는지 하나하나 탐색해야 되기때문에 속도가 느려질 수 있습니다.

#### 1. delete와 성능 저하 (피드백: 80점)<br/>
내용: delete가 일관성을 깬다는 점과 해시맵(Dictionary mode)으로 돌아간다는 설명은 완벽해.<br/>

보완 포인트: 엔진 내부에서는 이를 "Dict-mode" 혹은 "Slow-mode" 객체로 전환되었다고 해. 히든 클래스 체인을 아예 포기하고, 속성 탐색을 위해 무거운 해시 테이블(Dictionary)을 다시 생성해야 하므로 메모리 오버헤드가 크다는 점을 강조하면 더 좋아.

### 히든 클래스와 프로토타입 체인의 관계
**"우리는 인스턴스의 속성에 접근할 때 히든 클래스를 통해 오프셋을 찾는다고 배웠습니다. 그런데 만약 인스턴스에 없는 속성을 호출해서 **프로토타입 체인(Prototype Chain)**을 타고 올라가야 하는 상황이라면, 히든 클래스 기반의 최적화는 어떻게 작동할까요? 이 과정에서 '인라인 캐싱'은 어떤 역할을 할 수 있을까요?"**

먼저 인라인 캐싱은 비슷한 구조의 객체들에서 offset을 기준으로 캐싱해놨다가 바로 꺼내는 방식입니다.
해당 객체의 구조를 기억해놨다가 체인에 따라서 올라갈때, 상위 프로토타입에 저장되어있는 hidden class가 있다면 해당 class에 캐싱되어있는 정보를 기반으로 바로 꺼내줄 것 같습니다.

#### 2. 히든 클래스와 프로토타입 체인 (피드백: 75점)<br/>
내용: 상위 프로토타입의 히든 클래스를 캐싱한다는 아이디어는 방향은 맞지만, 조금 더 정교할 필요가 있어.<br/>

보완 포인트: 인라인 캐싱은 '객체의 히든 클래스' + '속성의 위치'를 한 쌍으로 기억해. 프로토타입 체인의 경우, 인라인 캐싱은 '히든 클래스'뿐만 아니라 '프로토타입의 히든 클래스'까지도 함께 캐싱하여, 체인을 매번 타지 않고도 바로 상위 프로토타입의 오프셋을 읽어오도록 최적화해. 이걸 'Polymorphic IC(다형성 인라인 캐싱)'와 연결해


### 왜 '생성자'에서 초기화해야 하는가?

**"개발자들이 흔히 '객체 생성 시 생성자에서 속성을 선언하라'고 조언합니다. 단순히 '깔끔한 코드'를 위해서일까요? V8 엔진의 히든 클래스 전환(Transition) 관점에서, 생성자 내부에서의 선언이 왜 성능상 필수적인지 **'객체 쉐이프(Shape)'**라는 용어를 섞어서 설명해 보세요."**

아니요 히든클래스를 생성하려면 일관화된 속성 추가 방식이어야합니다. 
속성을 추가하는 순서를 뒤죽박죽하게 된다면 히든클래스가 복잡해질 뿐더러 해시맵 방식으로 교체될 수 있습니다.
그래서 생성자라는 객체 쉐이프를 정하고 선언하라고 이야기하는 것입니다.


#### 3. 왜 생성자에서 초기화해야 하는가? (피드백: 95점)<br/>
내용: "객체 쉐이프(Shape)를 정하고 선언한다"는 표현이 아주 날카로웠어!<br/>

보완 포인트: 완벽해. 여기서 "생성자에서 속성을 선언하면 엔진은 그 객체 쉐이프를 '고정(Freeze)'하거나 '예측 가능'하게 인지하여, 객체 생성 직후부터 모노모픽(Monomorphic, 단형성) 상태의 인라인 캐싱을 즉시 적용할 수 있기 때문"이라고 한 문장만 덧붙이면 면접관은 "이 친구는 엔진 내부를 다 뜯어봤구나"라고 생각할 거야.
