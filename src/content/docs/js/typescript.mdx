## 타입 호환성

TypeScript에서는 타입 이름이 달라도, 필요한 속성과 메서드를 가지고 있으면 같은 타입처럼 사용할 수 있다.

```typescript
interface Food {
  name: string;
  caloriePer100g: number;
  weightInGram: number;
}

function calcCalorie(food: Food) {
  return (food.caloriePer100g * food.weightInGram) / 100;
}

// 1. Food 타입과 정확히 일치
const apple: Food = {
  name: "사과",
  caloriePer100g: 52,
  weightInGram: 150,
};

calcCalorie(apple); // 가능

// 2. Food 타입과 전혀 다른 구조
interface Computer {
  brand: string;
  cpu: string;
  ramGB: number;
}

const myComputer: Computer = {
  brand: "LG",
  cpu: "Intel i7",
  ramGB: 16,
};

calcCalorie(myComputer); // 에러

// 3. Food와 일부 속성만 일치
interface Burger {
  name: string;
  caloriePer100g: number;
}

const burger: Burger = {
  name: "치즈버거",
  caloriePer100g: 295,
};

calcCalorie(burger); // 에러

// 4. Food라고 선언하지 않았지만 구조가 같음
const burgerWithWeight = {
  name: "치즈버거",
  caloriePer100g: 295,
  weightInGram: 220,
};

calcCalorie(burgerWithWeight); // 가능

// 5. Food보다 속성이 더 많음
const chickenBurger = {
  name: "치킨버거",
  caloriePer100g: 260,
  weightInGram: 240,
  protein: 28,
  price: 6500,
};

calcCalorie(chickenBurger); // 가능
```

#### Naver의 설명

Typescript의 타입 검사란 `어떤 심벌에 대한 각종 대입/참조/연산이 가능한지 확인하는 과정`이라고 한다.<br/>
구조적 맥락 아래에서 가능한 타입을 추론하는데 그 과정에서 너무 많은 조합이 생길 수 있어서 그리디한 알고리즘으로 구현해서 타입을 추론하는 것 같다고 이야기한다.<br/>
여기서 제약 충족 문제(Constraint Satisfaction Problem)라는 개념도 나오고 NP-완전 문제도 나온다. <br/>

자세히 설명하자면,<br/>
제약 충족 문제는

```typescript
function identity<T>(value: T): T {
  return value;
}

const result = identity("hello");

// value의 실제 값은 "hello"다.
// "hello"는 string이다.
// 함수의 매개변수 타입은 T다.
// 반환 타입도 T다.
// 따라서 T는 string으로 볼 수 있다.
```

이렇게 여러 조건을 동시에 만족하는 답을 찾는 문제를 제약 충족 문제라고 한다. <br/>
하지만 여러가지 조건을 동시에 찾는 NP-완전 을 해결하는 것이 불가능하기 때문에 그리디한 알고리즘으로 구현해서 타입을 추론하는 것 같다고 이야기 하는 것 같다. <br/>

Typescript에서는 유연성만을 생각 할 순 없다. <br/>
따라서 어떤 경우에 호환이 되고 안되는지를 규칙이 필요하다. <br/>
그 규칙을 만드는 방법은 `명목적 서브타이핑(Nominal Subtyping)`과 `구조적 타이핑(Structural Typing)`이 존재한다.

### 1. 명목적 서브타이핑(Nominal Subtyping)

<pre>
  “네가 진짜로 그 타입이라고 선언됐어?” <br />
  “그 타입을 상속하거나 구현한다고 명시했어?”
</pre>

```typescript
interface Food {
  name: string;
  caloriePer100g: number;
  weightInGram: number;
}

interface Burger {
  name: string;
  caloriePer100g: number;
  weightInGram: number;
}

const burger: Burger = {
  name: "치즈버거",
  caloriePer100g: 295,
  weightInGram: 220,
};

const food: Food = burger; // 가능
```

이렇게 하면 타입 오류를 방지할 뿐더러 명확한 의도를 반영할 수 있다.

### 2. 구조적 타이핑(Structural Typing)

구조적 타이핑은 값이 가진 구조를 기준으로 타입 호환성을 판단하는 방식이다.<br/>
이름이나 명시적 선언이 필요 없다.

```typescript
interface Food {
  name: string;
  caloriePer100g: number;
  weightInGram: number;
}

function calcCalorie(food: Food) {
  return (food.caloriePer100g * food.weightInGram) / 100;
}

interface Burger {
  name: string;
  caloriePer100g: number;
  weightInGram: number;
  hasCheese: boolean;
}

const burger: Burger = {
  name: "치즈버거",
  caloriePer100g: 295,
  weightInGram: 220,
  hasCheese: true,
};

calcCalorie(burger); // 가능
```

  오리라고 부를 것이다.

### TypeScript의 타입 호환성은 구조적 타이핑을 기반으로 한다. <br/>

### 타입 호환 테스트

```typescript
const burger = {
  protein: 29,
  carbohydrates: 48,
  fat: 13,
  burgerBrand: "버거킹",
};

const calorie = calculateCalorie(burger);
/** 타입검사결과 : 오류없음 (OK) */

const calorie = calculateCalorie({
  protein: 29,
  carbohydrates: 48,
  fat: 13,
  burgerBrand: "버거킹",
});
/** 타임검사결과 : 오류 (NOT OK)*/
```

위의 결과가 나오는 원인은? <br/>
저번에 포스팅한 [TypeScript의 타입 시스템](https://yeaseul7.github.io/js/typescript_type/)를 살펴보면 Object 부분에서 typescript 멍청한 컴파일러때문에 일어나는 일이 아닐까 싶었다. <br/>

다시 Typescript의 컴파일러가 하는일을 생각해보자 <br/>
-> abstract syntax tree(AST)로 변환하고, 타입을 추론하고, 타입을 체크한다. <br/>

실제로 타입 검사하는 코드를 살펴보자.<br/>
[checker.ts 코드](https://raw.githubusercontent.com/microsoft/TypeScript/main/src/compiler/checker.ts) <br/>

```typescript
const isPerformingExcessPropertyChecks =
  !(intersectionState & IntersectionState.Target) &&
  isObjectLiteralType(source) &&
  getObjectFlags(source) & ObjectFlags.FreshLiteral;
if (isPerformingExcessPropertyChecks) {
  if (
    hasExcessProperties(source as FreshObjectLiteralType, target, reportErrors)
  ) {
    if (reportErrors) {
      reportRelationError(
        headMessage,
        source,
        originalTarget.aliasSymbol ? originalTarget : target,
      );
    }
    return Ternary.False;
  }
}
```

위에 있는 코드가 원본 코드고

```typescript
/** 함수 매개변수에 전달된 값이 FreshLiteral인 경우 true가 됩니다. */
const isPerformingExcessPropertyChecks =
  getObjectFlags(source) & ObjectFlags.FreshLiteral;

if (isPerformingExcessPropertyChecks) {
  /** 이 경우 아래 로직이 실행되는데,
   * hasExcessProperties() 함수는
   * excess property가 있는 경우 에러를 반환하게 됩니다.
   * 즉, property가 정확히 일치하는 경우만 허용하는 것으로
   * 타입 호환을 허용하지 않는 것과 같은 의미입니다. */
  if (hasExcessProperties(source as FreshObjectLiteralType)) {
    reportError();
  }
}
/**
 * FreshLiteral이 아닌 경우 위 분기를 skip하게 되며,
 * 타입 호환을 허용하게 됩니다. */
```

이게 Toss 포스터에서 적힌 코드이다.

중요한것은 FreshLiteral인지 아닌지에 따라서 excess property가 있는지 없는지에 따라 에러를 반환하는 hasExcessProperties 함수가 작동하는 것이다. <br/>

### Fresh Literal..?

<pre>
  “이 객체를 지금 이 자리에서 직접 만들었나?” <br />
  “아니면 이미 변수에 담겨서 독립적인 타입으로 추론이 끝났나?”
</pre>

```typescript
//Fresh Literal
type Nutrition = {
  protein: number;
  carbohydrates: number;
  fat: number;
};

function calculateCalories(nutrition: Nutrition) {
  return (
    nutrition.protein * 4 + nutrition.carbohydrates * 4 + nutrition.fat * 9
  );
}

calculateCalories({
  protein: 29,
  carbohydrates: 48,
  fat: 13,
  burgerBrand: "버거킹",
});
```

Typescript 입장

<pre>
  “지금 Nutrition 타입에 맞춰 객체를 직접 작성했네.
  <br />
  그런데 Nutrition에 없는 burgerBrand가 들어있네? <br />
  이건 오타거나 잘못 넣은 속성일 수 있으니 에러를 내자.”
</pre>

그리고 개발자 입장에서도 코드를 보면 `burgerBrand`가 타입에 존재하는 속성이라고 착각할 수 있다. <br/>

하지만

```typescript
const burger = {
  protein: 29,
  carbohydrates: 48,
  fat: 13,
  burgerBrand: "버거킹",
};

calculateCalories(burger);
```

위의 경우 burger는 이미 변수에 할당되면서 자기 타입이 추론됐다.

<pre>
  “burger는 protein, carbohydrates, fat을 모두 가지고 있네. <br />
  Nutrition이 요구하는 구조를 만족하니까 통과.”
</pre>

그리고 burgerBrand가 있어도 통과하는 이유는<br/>
Nutrition에 맞춰서 방금 생성된 것으로 보지 않고 burger라는 독립적인 타입으로 추론이 끝났기 때문이다. <br/>

### type Assertion 하면 Freshness가 사라진다?

```typescript
calculateCalories({
  protein: 29,
  carbohydrates: 48,
  fat: 13,
  burgerBrand: "버거킹",
} as Nutrition);
```

위의 코드는 `이 객체를 Nutrition으로 취급해.`라는 것과 같다.<br/>
그러면 초과 속성 검사를 강하게 하지 않고 넘어갈 수 있다. <br/>

⚠️ 타입이 빠져도 type Assertion하면 Freshness가 사라지기 때문에 초과 속성 검사를 강하게 하지 않고 넘어가서 런타임 에러가 발생할 수 있다. <br/>

## AI 질문

### 질문 1. "TypeScript가 채택한 구조적 타이핑은 개발 속도를 높여주지만, 동시에 의도치 않은 타입 호환을 허용할 위험도 있습니다. 명목적 타이핑(Nominal Typing)의 장점을 TypeScript에서 흉내 내기 위해(예: Brand 패턴), 실무에서는 어떤 기법들을 활용하고 있을까요?"

모르겠습니다..

-> Branding 기법<br/>

```typescript
// 'Brand'를 추가하는 방법
type UserId = string & { __brand: "UserId" };

function getUserId(id: string): UserId {
  return id as UserId; // 강제로 타입 단언하여 생성
}

// 명목적 타이핑처럼 동작!
function processUser(id: UserId) {
  /* ... */
}

const rawId = "user_123";
processUser(rawId); // ❌ 에러! 그냥 string은 UserId가 아니야.
processUser(getUserId(rawId)); // ✅ 성공!
```

-> 핵심: 인터섹션(&)과 가상의 속성(\_\_brand)을 추가해서, 타입 시스템이 서로를 다른 타입으로 인식하게 만드는 방식이야. 이게 실무에서 "명목적 타이핑"을 구현하는 대표적인 방법이야!

50/100

AI 답변 : "`Branding(Brand 패턴)`을 활용합니다.
예를 들어, string 타입의 userId와 orderId가 있을 때, 겉모습은 똑같이 문자열이라 실수를 유발할 수 있습니다. 이때 `type UserId = string & { __brand: 'UserId' }`와 같이 인터섹션 타입과 고유 구분자를 사용하면, 타입 시스템이 서로 다른 타입으로 인식하게 되어 오용을 방지할 수 있습니다. 저는 이를 통해 명목적 타이핑의 안전성을 구조적 타이핑 환경에서도 구현합니다."

### 질문 2. "초과 속성 검사(Excess Property Check)는 개발자의 실수를 막아주는 훌륭한 안전장치입니다. 만약 TypeScript 컴파일러가 모든 객체에 대해 엄격한 초과 속성 검사를 수행한다면, 대규모 프로젝트의 빌드 타임과 개발 경험(DX)에는 어떤 변화가 생길까요?"

Excess Property Check를 모든 객체에 적용이 된다면 매번 타입을 명시해줘야해서 더 많은 공수가 들어가서 Typescript가 너무 번거롭고 귀찮아져서 잘 안쓰게 될 수 있을 것 같습니다. 또한 코드가 어마하게 길어짐에 따라 빌드타임도 길어질 것 같습니다. 컴파일러가 Fresh Liter 상황에서만 검사하는 이유는 추론을 했을때 타입을 허용해줬다가 런타임 오류가 날 위험이 크다던가 개발자들이 오해할만한 코드가 생성되었을때를 방지하기 위해서입니다.

95/100

AI 답변 : "모든 객체에 대해 엄격한 검사를 수행하면 개발자가 매번 모든 속성을 타입 정의에 맞춰 선언해야 하므로 개발 생산성(DX)이 크게 저하됩니다. 또한, 빌드 시 컴파일러가 모든 리터럴 객체 내부를 순회하며 타입을 비교해야 하므로 빌드 타임 비용이 기하급수적으로 증가할 것입니다. 따라서 컴파일러는 Fresh Literal인 경우에만 오타 방지를 위해 초과 속성 검사를 수행하여, 안전성과 유연성 사이의 최적의 타협점을 찾은 것이라고 생각합니다."

### 질문 3. "우리는 as를 통해 컴파일러의 검사를 우회할 수 있습니다. 하지만 이는 런타임 에러의 잠재적 위험을 안고 있죠. '타입 단언을 사용하는 것이 정당화되는 실무적인 상황'은 무엇이며, 이를 안전하게 사용하기 위해 타입 가드(Type Guard)를 어떻게 결합하는 것이 좋을까요?"

만약 외부 라이브러리를 가져와서 사용하게 될때 그안에 어떤 타입으로 구성되어있는지 알 수 없을때가 있습니다. 그럴때 as를 이용해서 검사를 우회하고는 합니다. 그리고 타입 가드로 해당 심볼이 어떤 타입을 가졌을것이라고 예측하여 안전한 코드를 작성하는것이 좋다고 생각합니다.

90/100

AI 답변 : "외부 API 응답처럼 타입 구조를 100% 확신할 수 없는 상황에서 타입 단언을 사용하곤 합니다. 하지만 단순히 단언으로 끝내는 것이 아니라, **'사용자 정의 타입 가드(User-defined type guards)'**를 반드시 결합합니다. is 키워드를 사용하여 런타임에 데이터 구조를 검증하는 함수를 작성하면, 컴파일러가 단언 이후의 타입까지 안전하게 추론할 수 있어 훨씬 견고한 코드가 됩니다."
